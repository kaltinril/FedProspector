# Phase 44.5: EF Core Navigation Properties

**Status**: DEFERRED
**Depends on**: Phase 44.3 (Data Enrichment Opportunities)

## Context

The C# API uses EF Core but many cross-table relationships lack navigation properties. Services resort to manual LEFT JOINs (LINQ query syntax), correlated subqueries, separate queries, or raw SQL where EF Core navigation properties should be used instead.

The correct EF Core approach is:
1. Define navigation properties on model classes
2. Configure relationships in `OnModelCreating` using `HasOne`/`HasMany`
3. Use `.Include()`/`.ThenInclude()` in queries

**Schema ownership rule**: Python DDL owns all ETL/data table schemas. EF Core maps to these existing tables without managing their schema. No EF migrations for ETL tables. No DB-level FK constraints on ETL tables -- only EF-level relationship mappings.

## Item 1: Vendor/Recipient Entity Navigation

**Source**: Phase 44.3 item 2.1
**Relationship type**: Optional HasOne (not all UEIs have matching entity records)

Three tables have vendor UEI columns that should navigate to the `Entity` model:

### 1a. FpdsContract.VendorUei -> Entity

- **Model**: `FpdsContract` (table: `fpds_contract`)
- **FK property**: `VendorUei` (VARCHAR(12))
- **Principal**: `Entity.UeiSam` (PK, VARCHAR(12))
- **Relationship**: `HasOne<Entity>().WithMany().HasForeignKey(c => c.VendorUei).IsRequired(false)`
- **Navigation property to add**: `public Entity? VendorEntity { get; set; }` on `FpdsContract`
- **No DB constraint**: Many vendor UEIs have no matching entity row (entities loaded on-demand)

**Current manual pattern** (AwardService.GetDetailAsync, lines 204-221):
```
var entity = await _context.Entities.AsNoTracking()
    .FirstOrDefaultAsync(e => e.UeiSam == contract.VendorUei);
```
Executed as a separate query after fetching the contract. With navigation property, this becomes `.Include(c => c.VendorEntity)`.

### 1b. UsaspendingAward.RecipientUei -> Entity

- **Model**: `UsaspendingAward` (table: `usaspending_award`)
- **FK property**: `RecipientUei` (VARCHAR(12))
- **Principal**: `Entity.UeiSam` (PK, VARCHAR(12))
- **Relationship**: `HasOne<Entity>().WithMany().HasForeignKey(a => a.RecipientUei).IsRequired(false)`
- **Navigation property to add**: `public Entity? RecipientEntity { get; set; }` on `UsaspendingAward`

**Current manual pattern** (AwardService.GetDetailAsync, lines 316-333):
Same separate query pattern -- fetches entity by `usaAward.RecipientUei` in a second round-trip.

### 1c. SamSubaward.SubUei -> Entity

- **Model**: `SamSubaward` (table: `sam_subaward`)
- **FK property**: `SubUei` (VARCHAR(12))
- **Principal**: `Entity.UeiSam` (PK, VARCHAR(12))
- **Relationship**: `HasOne<Entity>().WithMany().HasForeignKey(s => s.SubUei).IsRequired(false)`
- **Navigation property to add**: `public Entity? SubEntity { get; set; }` on `SamSubaward`

**Current manual pattern** (SubawardService): No entity lookup currently. Raw SQL returns `sub_name` and `sub_uei` without enrichment.

### 1d. SamSubaward.PrimeUei -> Entity

- **Model**: `SamSubaward` (table: `sam_subaward`)
- **FK property**: `PrimeUei` (VARCHAR(12))
- **Principal**: `Entity.UeiSam` (PK, VARCHAR(12))
- **Relationship**: `HasOne<Entity>().WithMany().HasForeignKey(s => s.PrimeUei).IsRequired(false)`
- **Navigation property to add**: `public Entity? PrimeEntity { get; set; }` on `SamSubaward`

**Effort**: ~0.5 days for all four. Model changes are trivial; service refactoring to use `.Include()` is the main work.

## Item 2: Opportunity to Award Chain

**Source**: Phase 44.3 item 3.1

### 2a. Opportunity.SolicitationNumber -> FpdsContract.SolicitationNumber

- **Model**: `Opportunity` (table: `opportunity`)
- **FK property**: `SolicitationNumber` (VARCHAR(100))
- **Principal key**: `FpdsContract.SolicitationNumber` (VARCHAR(100)) -- NOT a PK
- **Relationship type**: One-to-Many (one solicitation can produce multiple contracts/modifications)
- **Navigation property to add**: `public ICollection<FpdsContract> RelatedContracts { get; set; }` on `Opportunity`
- **Complication**: `SolicitationNumber` is not a unique key on `FpdsContract` (composite PK is ContractId + ModificationNumber). EF Core requires the principal key to be a PK or have a unique index. This may need to remain as a manual query.

**Current manual pattern** (OpportunityService.GetDetailAsync, lines 179-194):
```
await _context.FpdsContracts.AsNoTracking()
    .Where(c => c.SolicitationNumber == opp.SolicitationNumber && c.ModificationNumber == "0")
    .Select(c => new RelatedAwardDto { ... })
    .ToListAsync();
```
This query also filters by `ModificationNumber == "0"` (base awards only), which a navigation property cannot express. **Recommendation**: Keep this as a manual query OR define a navigation that loads all matching contracts and filter in the projection. The manual query is likely cleaner here.

### 2b. Opportunity.SolicitationNumber -> UsaspendingAward.SolicitationIdentifier

- **Same complication**: `SolicitationIdentifier` is not a key on `UsaspendingAward`.
- **Current manual pattern** (OpportunityService.GetDetailAsync, lines 220-237): Separate query by `SolicitationIdentifier`.

**Effort**: ~0.25 days if implemented as navigations; may decide to keep manual queries due to non-key join columns and filtering requirements. Investigation needed.

**Decision needed**: Whether the non-key join is worth modeling as a formal EF relationship or whether manual queries are more appropriate here.

## Item 3: Award to Subaward Chain

**Source**: Phase 44.3 item 3.2

### 3a. SamSubaward.PrimePiid -> FpdsContract.ContractId

- **Model**: `SamSubaward` (table: `sam_subaward`)
- **FK property**: `PrimePiid` (VARCHAR(50))
- **Principal key**: `FpdsContract.ContractId` -- part of composite PK (ContractId, ModificationNumber)
- **Relationship type**: Many-to-Many (one PIID has multiple modifications; one PIID can have multiple subawards)
- **Complication**: `ContractId` is only half of `FpdsContract`'s composite PK. EF Core does not support navigation properties where the FK maps to only part of a composite key.
- **Navigation property to add**: `public ICollection<SamSubaward> Subawards { get; set; }` on `FpdsContract` -- but this requires `ContractId` to be a standalone key or alternate key, which it is not.

**Current manual pattern** (SubawardService.GetSubawardsByPrimeContractAsync, lines 132-167):
Raw ADO.NET query: `WHERE prime_piid = @primePiid`. No EF Core LINQ at all.

**Recommendation**: Define `ContractId` as an alternate key on `FpdsContract` if feasible, then configure `HasMany<SamSubaward>().WithOne().HasForeignKey(s => s.PrimePiid).HasPrincipalKey(c => c.ContractId)`. Alternatively, keep as manual query and refactor SubawardService to use LINQ instead of raw SQL.

**Effort**: ~0.5 days. Includes investigating the alternate key approach and refactoring SubawardService from raw ADO.NET to LINQ.

## Item 4: Reference Table Navigations

**Source**: Phase 44.3 items 1.4 and 2.4

Currently, reference table lookups are done via manual LEFT JOINs in LINQ query syntax or separate queries. These should be navigation properties where the join key is simple (single-column, matches PK).

### 4a. NAICS Code (simple -- single PK)

**Tables using NAICS**: `FpdsContract.NaicsCode`, `UsaspendingAward.NaicsCode`, `SamSubaward.NaicsCode`, `Opportunity.NaicsCode`
**Reference model**: `RefNaicsCode` (PK: `NaicsCode` VARCHAR(11))
**Relationship**: `HasOne<RefNaicsCode>().WithMany().HasForeignKey(x => x.NaicsCode).IsRequired(false)`
**Navigation property**: `public RefNaicsCode? NaicsRef { get; set; }`

**Current manual pattern** (used in all three services):
- AwardService.SearchAsync (lines 89-94): LINQ query-syntax LEFT JOIN `join n in _context.RefNaicsCodes on c.NaicsCode equals n.NaicsCode`
- AwardService.GetDetailAsync (lines 229-233): Separate query `_context.RefNaicsCodes.FirstOrDefaultAsync(...)`
- OpportunityService.SearchAsync (lines 61-63): Same LEFT JOIN pattern
- OpportunityService.GetDetailAsync (lines 143-145): Separate query
- EntityService.GetDetailAsync (lines 133-144): LEFT JOIN on `EntityNaics`

### 4b. Set-Aside Type (simple -- single PK)

**Tables using set-aside**: `FpdsContract.SetAsideType`, `Opportunity.SetAsideCode`
**Reference model**: `RefSetAsideType` (PK: `SetAsideCode` VARCHAR(10))
**Relationship**: `HasOne<RefSetAsideType>().WithMany().HasForeignKey(x => x.SetAsideType).IsRequired(false)`
**Navigation property**: `public RefSetAsideType? SetAsideRef { get; set; }`

**Note**: `UsaspendingAward.TypeOfSetAside` uses a different column name but same reference table. FK configuration must specify the correct property.

**Current manual pattern**:
- AwardService.SearchAsync (lines 89-94): LINQ JOIN
- AwardService.GetDetailAsync (lines 245-251): Separate query
- OpportunityService.SearchAsync (lines 61-66): LINQ JOIN
- OpportunityService.GetDetailAsync (lines 153-159): Separate query

### 4c. PSC Code (BLOCKED -- composite PK)

**Reference model**: `RefPscCode` (composite PK: `PscCode` + `StartDate`)
**Problem**: The FK from data tables is `PscCode` alone, but the reference table has a composite PK including `StartDate`. One PSC code can have multiple rows with different effective dates. Services currently use `OrderByDescending(p => p.StartDate).FirstOrDefault()` to get the most recent description.

**This CANNOT be a navigation property.** EF Core requires FK to match the full PK (or a unique alternate key). Since there are multiple rows per PSC code, there is no 1:1 mapping.

**Recommendation**: Keep as manual queries. The current pattern of correlated subqueries works:
```
_context.RefPscCodes.Where(p => p.PscCode == x.PscCode)
    .OrderByDescending(p => p.StartDate).Select(p => p.PscName).FirstOrDefault()
```

### 4d. Business Type (simple -- single PK)

**Used by**: `EntityBusinessType.BusinessTypeCode`
**Reference model**: `RefBusinessType` (PK: `BusinessTypeCode` VARCHAR(4))
**Relationship**: `HasOne<RefBusinessType>().WithMany().HasForeignKey(bt => bt.BusinessTypeCode).IsRequired(false)`
**Navigation property**: `public RefBusinessType? BusinessTypeRef { get; set; }` on `EntityBusinessType`

**Current manual pattern** (EntityService.GetDetailAsync, lines 159-168): LINQ LEFT JOIN.

### 4e. State Code (BLOCKED -- composite PK)

**Reference model**: `RefStateCode` (composite PK: `StateCode` + `CountryCode`)
**Problem**: Data tables store only state codes (e.g., "VA"), not country codes. The reference table has composite PK (state_code, country_code). Services currently filter with `s.CountryCode == "USA"` as a hardcoded assumption.

**This CANNOT be a simple navigation property.** Would require matching on both StateCode AND CountryCode, but data tables only store state code.

**Recommendation**: Keep as manual queries with the `CountryCode == "USA"` filter. Current pattern in services:
```
_context.RefStateCodes.FirstOrDefaultAsync(s => s.StateCode == x.PopState && s.CountryCode == "USA")
```

### 4f. Country Code (simple -- single PK)

**Tables using country**: `FpdsContract.PopCountry`, `UsaspendingAward.PopCountry`, `SamSubaward.PopCountry`, `Opportunity.PopCountry`, `EntityAddress.CountryCode`
**Reference model**: `RefCountryCode` (PK: `ThreeCode` VARCHAR(3))
**Relationship**: `HasOne<RefCountryCode>().WithMany().HasForeignKey(x => x.PopCountry).IsRequired(false)`
**Navigation property**: `public RefCountryCode? PopCountryRef { get; set; }`

**Note**: `EntityAddress` uses `CountryCode` as the property name, not `PopCountry`. Separate configuration needed.

**Current manual pattern**: Separate queries in AwardService, OpportunityService, EntityService detail methods.

### Summary of Reference Table Navigations

| Ref Table | PK Type | Can Be Nav Property? | Models to Update |
|-----------|---------|---------------------|------------------|
| RefNaicsCode | Simple | YES | FpdsContract, UsaspendingAward, SamSubaward, Opportunity |
| RefSetAsideType | Simple | YES | FpdsContract, Opportunity, UsaspendingAward |
| RefPscCode | Composite | NO -- keep manual | (none) |
| RefBusinessType | Simple | YES | EntityBusinessType |
| RefStateCode | Composite | NO -- keep manual | (none) |
| RefCountryCode | Simple | YES | FpdsContract, UsaspendingAward, SamSubaward, Opportunity, EntityAddress |

**Effort**: ~1 day. Four reference tables get navigation properties. Add nav properties to ~12 model classes. Update DbContext with ~12 relationship configurations. Refactor services to use `.Include()` where beneficial (some JOINs in search queries may stay as LINQ syntax for projection efficiency).

## Item 5: Contracting Office to Federal Hierarchy

**Source**: Phase 44.3 item 2.2

### 5a. Opportunity.ContractingOfficeId -> FederalOrganization.AgencyCode

- **Model**: `Opportunity` (table: `opportunity`)
- **FK property**: `ContractingOfficeId` (VARCHAR(20))
- **Principal**: `FederalOrganization.AgencyCode` (VARCHAR(20)) -- NOT the PK (PK is `FhOrgId` INT)
- **Relationship**: Would need `HasPrincipalKey(fo => fo.AgencyCode)` to use AgencyCode as alternate key
- **Complication**: `AgencyCode` is NOT unique in `federal_organization` -- multiple org levels can share an agency code. This cannot be a HasOne navigation without a unique alternate key.

**Current status**: No lookup exists in any service. `Opportunity.DepartmentName`, `SubTier`, and `Office` are stored as denormalized strings. The contracting_office_id is populated (14,240/14,241 rows) but never resolved.

**Blocked until**: Investigation of code matching (contracting_office_id vs agency_code vs oldfpds_office_code) is complete, AND office-level organizations are loaded by the Python ETL. May also require a unique index on the appropriate column in `federal_organization`.

**Effort**: ~0.5 days investigation + 0.25 days implementation (if feasible).

## What NOT to Change

1. **No DB-level FK constraints on ETL tables.** Python DDL owns schema for `fpds_contract`, `usaspending_award`, `sam_subaward`, `opportunity`, `entity`, and all `ref_*` tables. EF Core maps relationships for query convenience only -- `HasForeignKey()` without a matching DB constraint is valid and supported.

2. **No EF migrations for ETL tables.** All schema changes to data/ETL tables go through Python DDL in `fed_prospector/db/schema/`.

3. **Do not add `[ForeignKey]` data annotations.** Use Fluent API in `OnModelCreating` exclusively, consistent with existing patterns in `FedProspectorDbContext`.

4. **Do not add inverse collection navigations on `Entity` or `RefNaicsCode` etc. unless needed.** Adding `ICollection<FpdsContract> Contracts` to `Entity` would mean EF tracks those collections. Use `.WithMany()` without a navigation property parameter (one-way navigation) to keep models lean.

5. **Do not replace efficient search query patterns.** Some search methods use LINQ query syntax with manual JOINs for good reason -- projecting directly to DTOs without materializing full entities. Navigation properties with `.Include()` materialize the full related entity. For search/list endpoints, the current projection patterns may be more efficient. Navigation properties are most valuable for detail endpoints where full related data is needed.

6. **PSC Code and State Code lookups stay manual.** Composite PKs prevent clean navigation property mapping. The current patterns work correctly.
