# Phase 44.10: Entity POC Normalization

**Status**: PLANNED
**Priority**: Medium — data bloat and maintenance issue, not blocking
**Discovered**: Phase 44.6 validation, DB size audit

## Problem Statement

The `entity_poc` table stores denormalized point-of-contact data per entity. The same person (e.g., "Teri Hamilton") appears in hundreds of rows with identical name, email, phone, and address — once per entity they're associated with. This causes:

- **Table bloat**: `entity_poc` is larger than the `entity` table itself (~750K+ rows of mostly duplicate people)
- **Update pain**: Changing a contact's phone number requires updating every row
- **No single view of a person**: Can't answer "what entities is Teri associated with?" without scanning duplicates

## Current Schema

```
entity_poc (denormalized — one row per entity+person+poc_type)
  poc_id, entity_id, poc_type, first_name, last_name, middle_initial,
  title, phone, phone_ext, email, address_*, ...
```

Same person duplicated N times across N entities (or even multiple poc_types on the same entity).

## Proposed Schema

```
contact (normalized — one row per unique person)
  contact_id  PK
  full_name, first_name, last_name, middle_initial
  title, email, phone, phone_ext
  address_line1, address_line2, city, state_code, zip_code, country_code
  created_at, updated_at
  UNIQUE INDEX (full_name(100), email(100))   -- natural key for dedup

entity_contact (M:M junction — links entities to contacts)
  id  PK
  entity_id   FK → entity
  contact_id  FK → contact
  poc_type    VARCHAR(20)   -- GOVT_BUSINESS_POC, ELECTRONIC_BUSINESS_POC, etc.
  source_load_date  DATETIME  -- when this link was last confirmed by a SAM load
  UNIQUE INDEX (entity_id, contact_id, poc_type)
```

## Advantages

- **Massive dedup**: Teri Hamilton stored once, referenced 6+ times via junction rows
- **Single update point**: Change Teri's phone in one row, all entity references reflect it
- **Table size reduction**: Contact table will be a fraction of current entity_poc size
- **Enables contact-level features**: Contact history, relationship tracking, cross-entity POC view
- **Cleaner data model**: Matches how contracting_officer / opportunity_poc already work

## Disadvantages

- **Stale link detection is harder**: When SAM.gov changes an entity's POC from Teri to Bob, the loader must:
  1. Compare incoming POC list against existing junction rows for that entity
  2. Remove stale links (entity no longer lists Teri as that poc_type)
  3. Add new links (entity now lists Bob)
  4. Decide what to do with orphaned contacts (Teri with 0 remaining references — keep? soft-delete? purge?)
  - With denormalized data, you just overwrite the row. With normalized data, it's a diff operation.
- **Natural key ambiguity**: Two different "John Smith" people may collide on `(full_name, email)` if email is NULL. Need a strategy for NULL-email contacts.
- **Migration complexity**: Existing 750K+ rows need ETL'd into the new schema — extract unique contacts, build junction rows, drop old table
- **Loader complexity increases**: `entity_loader.py` currently does simple INSERT/UPDATE per POC row. Normalized version needs: lookup-or-create contact, then upsert junction row.
- **C# API impact**: Need to verify if the API reads entity_poc directly or through a view/service
