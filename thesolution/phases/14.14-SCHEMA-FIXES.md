# Phase 14.14 — Database Schema Fixes

## Status: COMPLETE (2026-03-03)

---

## Goal

Code review and schema audit found missing indexes, a broken multi-tenant uniqueness constraint, DDL sync gaps, and several data integrity issues. This phase fixes them with migrations and DDL corrections. No application code changes except where the schema fix requires it.

---

## CRITICAL Fixes

### Fix 1 — `prospect` UNIQUE KEY Breaks Multi-Tenancy

**File:** `fed_prospector/db/schema/tables/60_prospecting.sql:67`

**Problem:** `UNIQUE KEY uk_prospect_notice (notice_id)` constrains a notice to exactly one prospect in the entire database. In a multi-org SaaS deployment, if Org A creates a prospect for notice `ABC123`, Org B's attempt to track the same opportunity fails with a duplicate key error. This is a hard constraint failure for the product's core use case.

**Fix:** Change the unique constraint to scope to the organization:
```sql
-- Migration:
ALTER TABLE prospect
  DROP INDEX uk_prospect_notice,
  ADD UNIQUE KEY uk_prospect_org_notice (organization_id, notice_id);
```
Update `60_prospecting.sql` to reflect this in the baseline DDL.

### Fix 2 — `opportunity` Baseline DDL Missing 7 Phase 9 Columns

**File:** `fed_prospector/db/schema/tables/30_opportunity.sql`

## HIGH Fixes

### Fix 3 — Missing Index on `entity_disaster_response.uei_sam`

**File:** `fed_prospector/db/schema/tables/20_entity.sql:148-159`

**Problem:** `entity_loader.py` runs `DELETE FROM entity_disaster_response WHERE uei_sam = %s` on every entity update. Without an index on `uei_sam`, this is a full table scan. All other entity child tables (`entity_address`, `entity_naics`, `entity_psc`, `entity_business_type`, `entity_sba_certification`, `entity_poc`) have indexes on their `uei_sam` column. `entity_disaster_response` is the sole exception.

**Fix:**
```sql
-- Migration:
ALTER TABLE entity_disaster_response ADD INDEX idx_edr_uei (uei_sam);
```
Add `INDEX idx_edr_uei (uei_sam)` to the CREATE TABLE in `20_entity.sql`.

### Fix 4 — Missing Index on `usaspending_award.last_modified_date`

**File:** `fed_prospector/db/schema/tables/70_usaspending.sql`

**Problem:** `idx_usa_dates` indexes `(start_date, end_date)` but not `last_modified_date`. Incremental loads that filter by modification date (a core pattern) perform full table scans.

**Fix:**
```sql
ALTER TABLE usaspending_award ADD INDEX idx_usa_modified (last_modified_date);
```

### Fix 5 — Missing Index on `fpds_contract.ultimate_completion_date`

**File:** `fed_prospector/db/schema/tables/40_federal.sql`

**Problem:** Re-compete prediction queries filter by `ultimate_completion_date` to find contracts expiring soon. There is an index on `completion_date` (`idx_fpds_completion`) but not on `ultimate_completion_date`. The `v_procurement_intelligence` view uses `COALESCE(ultimate_completion_date, completion_date)` — queries that filter on this computed value can't use either index.

**Fix:**
```sql
ALTER TABLE fpds_contract ADD INDEX idx_fpds_ultimate_completion (ultimate_completion_date);
```

### Fix 6 — Missing Index on `fpds_contract.idv_piid`

**File:** `fed_prospector/db/schema/tables/40_federal.sql`

**Problem:** `idv_piid` links task orders to their parent IDV contract. Queries that find all task orders under an IDV (a common competitive intelligence query) do full table scans.

**Fix:**
```sql
ALTER TABLE fpds_contract ADD INDEX idx_fpds_idv_piid (idv_piid(50));
```
Use a prefix index since `idv_piid` is VARCHAR(100).

### Fix 7 — `sam_exclusion` Has No UNIQUE Constraint

**File:** `fed_prospector/db/schema/tables/40_federal.sql:112-137`

**Problem:** `sam_exclusion` has no unique key on its natural business key. If the loader crashes between delete and re-insert, or if the same exclusion appears under slightly different raw JSON, duplicate rows accumulate silently. `INSERT ... ON DUPLICATE KEY UPDATE` cannot be used without a unique key to detect duplicates on.

**Fix:**
```sql
ALTER TABLE sam_exclusion
  ADD UNIQUE KEY uk_exclusion_natural (uei(50), exclusion_type(50), activation_date);
```
Note: `uei` can be NULL for entity-name-only exclusions. For those, use a generated hash column or accept that NULL UEIs don't participate in the uniqueness constraint (MySQL treats NULLs as distinct in UNIQUE KEYs — multiple NULL-UEI rows for the same exclusion type and date will be allowed). Document this as a known data quality limitation.

### Fix 8 — `record_hash` Type Inconsistency: `VARCHAR(64)` in `usaspending_award`

**File:** `fed_prospector/db/schema/tables/70_usaspending.sql:51`

**Problem:** All tables use `record_hash CHAR(64)` (SHA-256 always produces exactly 64 hex chars). `usaspending_award` alone uses `VARCHAR(64)`. This is a maintenance hazard and wastes storage (VARCHAR has 1-2 byte length overhead vs. CHAR's fixed width).

**Fix:**
```sql
ALTER TABLE usaspending_award MODIFY COLUMN record_hash CHAR(64) NULL;
```

---

### Fix 9 — `entity_address` Case Mismatch Between Loader and CLI

**Files:**
- Loader (`entity_loader.py:434`): inserts `address_type = 'PHYSICAL'` (uppercase)
- CLI (`entities.py:415`): queries `WHERE address_type = 'physical'` (lowercase)

**Problem:** Works by accident if the DB collation is case-insensitive (`utf8mb4_unicode_ci`), but fails silently if collation is ever changed. The correct value per the loader is `'PHYSICAL'`.

**Fix:** Update `entities.py:415` to use `'PHYSICAL'` (uppercase), matching the loader's insert value. No schema change needed.

## MEDIUM Fixes

### Fix 10 — `usaspending_transaction` NULL `modification_number` Allows Duplicates

**File:** `fed_prospector/db/schema/tables/70_usaspending.sql`

**Problem:** `UNIQUE KEY uk_txn_dedup (award_id, modification_number, action_date)` includes nullable `modification_number`. MySQL treats two NULL values as distinct in a UNIQUE KEY, so transactions with `NULL` modification_number can be inserted multiple times. `INSERT IGNORE` in the loader won't deduplicate them.

**Fix:** Make `modification_number` NOT NULL with an empty-string default:
```sql
ALTER TABLE usaspending_transaction
  MODIFY COLUMN modification_number VARCHAR(20) NOT NULL DEFAULT '';
```
Update the loader to insert `''` instead of `NULL` for missing modification numbers.

---

### Fix 11 — `ref_psc_code` Nullable Column in PRIMARY KEY

**File:** `fed_prospector/db/schema/tables/10_reference.sql:41-58`

**Problem:** PRIMARY KEY is `(psc_code, start_date)` but `start_date DATE` has no `NOT NULL` constraint. MySQL permits NULL in a PK column but the behavior is semantically undefined and can cause duplicate detection issues.

**Fix:**
```sql
ALTER TABLE ref_psc_code MODIFY COLUMN start_date DATE NOT NULL DEFAULT '1970-01-01';
```

---

### Fix 12 — `sam_exclusion.entity_name` Index Silently Truncated

**File:** `fed_prospector/db/schema/tables/40_federal.sql`

**Problem:** `entity_name VARCHAR(500)` with `INDEX idx_excl_entity_name (entity_name)` — with `utf8mb4` encoding (4 bytes/char), this index exceeds InnoDB's 767-byte prefix limit. MySQL silently truncates the index to the first 191 characters. Searches on long entity names may produce silent false negatives.

**Fix:**
```sql
ALTER TABLE sam_exclusion
  DROP INDEX idx_excl_entity_name,
  ADD INDEX idx_excl_entity_name (entity_name(100));
```
