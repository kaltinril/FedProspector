## Findings from Test Build

**Test database**: `fed_contracts_test` (still exists, can be reused)

**Fresh build output**: `56 tables, 2 views` (production has 57 tables + 4 views)

Exact discrepancies (prod vs test):

| What | Production | Test | Missing from test |
|------|-----------|------|-------------------|
| Tables | 57 | 56 | `usaspending_transaction` |
| Views | 4 | 2 | `v_incumbent_profile`, `v_procurement_intelligence` |
| EF internal | 1 | 0 | `__efmigrationshistory` (created by EF Core, not DDL) |

### Root cause of missing views

**Bug in `build-database` command** (`fed_prospector/cli/database.py` line ~161):

```python
if stmt and not stmt.startswith("--") and not stmt.startswith("USE "):
```

The `_split_sql_statements` function splits on `;`. After `USE fed_contracts;`, the next chunk is:
```
-- Procurement intelligence: joins opportunity -> fpds_contract -> usaspending_award
-- for a given solicitation. Returns award history, bidder count, burn rate, incumbent.
CREATE OR REPLACE VIEW v_procurement_intelligence AS
SELECT ...
```

Since the chunk starts with `--`, the `startswith("--")` check skips the **entire block** including the CREATE VIEW statement. This affects `v_procurement_intelligence` and `v_incumbent_profile` — both have comments between the `USE` and the `CREATE VIEW`.

`v_target_opportunities` and `v_competitor_analysis` work because their comments are before the `USE` statement, so the comment ends up in the first (USE) chunk which is also skipped, leaving the CREATE VIEW in its own clean chunk.

**Fix**: Strip leading comment lines from each statement before checking `startswith("--")`.

## EF Core Strategy Post-V1

When V1 ships and we need incremental schema changes:
1. The snapshot (`FedProspectorDbContextModelSnapshot.cs`) captures the current model state
2. Generate new migrations with `dotnet ef migrations add <Name> --project src/FedProspector.Infrastructure --startup-project src/FedProspector.Api`
3. Review generated migration (trim any drift from Python-owned tables)
4. Apply with `dotnet ef database update`
5. Also update the Python DDL scripts so fresh builds remain in sync

## Notes

- `dotnet-ef` tool and `Microsoft.EntityFrameworkCore.Design` package are pinned to 9.0.7 to match Pomelo 9.0.0 (no Pomelo 10.x available yet — tracked at https://github.com/PomeloFoundation/Pomelo.EntityFrameworkCore.MySql/issues/2007)
- MySQL 8.4 does NOT support `ADD COLUMN IF NOT EXISTS` — that's MariaDB-only syntax
- The `build-database` CLI command skips `USE` statements, so `DB_NAME` env var controls which database is targeted
