## Context

The schema has 7 staging tables designed to preserve raw API JSON before normalization. The architecture comment in `80_raw_staging.sql` explicitly states the intent:

> Enables re-processing/replay without re-fetching from APIs, capturing fields not yet normalized, historical audit trail, schema evolution safety.

**None of the 7 loaders write to their staging table.** All go directly: API response → normalize → upsert to production. The staging tables are empty despite 865k entities, 12k+ awards, and other data being loaded.

This was confirmed by grepping all Python files — only `db_maintenance.py` references `stg_entity_raw` (a cleanup routine), but since nothing writes there it does nothing.

The immediate trigger: 131 SAM.gov API calls (key 2 = 1,000/day limit) loaded 12,423 awards. If the normalizer has a bug or `fpds_contract` needs a new column, the data must be re-pulled from the API, burning quota.

## Implementation Pattern

### Transaction Isolation (Critical)

All loaders use a main connection with `autocommit=False` and batch commits every 500 records. On per-record error they call `conn.rollback()`. If staging writes share this connection, they would be rolled back on normalization errors — defeating the purpose.

**Solution**: Use a second connection with `autocommit=True` for staging writes only. One staging connection per load call, opened at the top and closed at the end.

```python
conn = get_connection()              # main connection (normalization + production upsert)
stg_conn = get_connection()          # staging connection (autocommit=True, write-once per record)
stg_cursor = stg_conn.cursor()
```

The `get_pool()` returns a pool of 5 connections; this temporarily uses 2 from the pool during a load. Acceptable.

### Per-Loader: Two Private Methods

**`_insert_staging(stg_cursor, load_id, key_vals: dict, raw: dict) -> int`**

Computes `raw_record_hash` from the raw JSON, inserts to staging, returns `lastrowid`.

```python
import hashlib, json as _json

raw_str = _json.dumps(raw, sort_keys=True, default=str)
raw_hash = hashlib.sha256(raw_str.encode()).hexdigest()
stg_cursor.execute(
    "INSERT INTO stg_fpds_award_raw "
    "(load_id, contract_id, modification_number, raw_json, raw_record_hash) "
    "VALUES (%s, %s, %s, %s, %s)",
    (load_id, key_vals["contract_id"], key_vals.get("modification_number"), raw_str, raw_hash)
)
return stg_cursor.lastrowid
```

**`_mark_staging(stg_cursor, staging_id, processed: str, error_msg=None)`**

Updates the outcome after normalization attempt.

```python
stg_cursor.execute(
    "UPDATE stg_fpds_award_raw SET processed=%s, error_message=%s WHERE id=%s",
    (processed, error_msg[:500] if error_msg else None, staging_id)
)
```

`processed` values: `'Y'` = success, `'E'` = error, `'N'` = default (unchanged records that skip production write are also marked `'Y'` — they were valid, just not re-written).

### Modified Main Loop

```python
stg_conn = get_connection()
stg_conn.autocommit = True
stg_cursor = stg_conn.cursor()
conn = get_connection()
cursor = conn.cursor(dictionary=True)
try:
    for raw in data:
        stats["records_read"] += 1
        staging_id = None
        record_key = None
        try:
            # --- staging write (committed immediately via stg_conn) -----
            key_vals = self._extract_staging_key(raw)
            staging_id = self._insert_staging(stg_cursor, load_id, key_vals, raw)

            # --- normalize -------------------------------------------
            normalized = self._normalize(raw)
            record_key = ...

            # --- change detection ------------------------------------
            new_hash = self.change_detector.compute_hash(normalized, _HASH_FIELDS)
            normalized["record_hash"] = new_hash
            if existing_hashes.get(record_key) == new_hash:
                stats["records_unchanged"] += 1
                self._mark_staging(stg_cursor, staging_id, 'Y')
                continue

            # --- upsert to production --------------------------------
            outcome = self._upsert(cursor, normalized, load_id)
            ...
            self._mark_staging(stg_cursor, staging_id, 'Y')

        except Exception as rec_exc:
            stats["records_errored"] += 1
            if staging_id:
                self._mark_staging(stg_cursor, staging_id, 'E', str(rec_exc))
            conn.rollback()
            batch_count = 0
            continue
        ...
    conn.commit()
finally:
    cursor.close()
    conn.close()
    stg_cursor.close()
    stg_conn.close()
```

## Entity Loader Special Case

`entity_loader.py` has two entry points:

- **`load_from_json_extract(file_path)`** — Reads from disk files (monthly/daily SAM extract). The file IS the raw data. **Do NOT add staging write here.** Files are already persisted.
- **`load_from_api_response(entity_data_list)`** — From API. **Add staging write here.**

The staging write goes into `_process_entities()` but only when called from the API path. Pass a flag: `_process_entities(iter, load_id, write_staging=False)` — set to `True` from `load_from_api_response`, `False` from `load_from_json_extract`.

### stg_entity_raw Schema Difference

`stg_entity_raw` (in `20_entity.sql`) predates the new staging tables and has a slightly different schema:
- **No auto-increment `id`** — just indexed, not a PK. Use `uei_sam + load_id` to track rows.
- **`processed_at DATETIME`** — extra column for when it was processed. Set to `NOW()` when marking processed.
- **`raw_json JSON`** (nullable) and `raw_record_hash CHAR(64)` (nullable) — unlike the newer tables which are NOT NULL.

Entity staging INSERT:
```sql
INSERT INTO stg_entity_raw (load_id, uei_sam, raw_json, raw_record_hash)
VALUES (%s, %s, %s, %s)
```

Entity staging update uses `uei_sam + load_id` instead of `id`:
```sql
UPDATE stg_entity_raw SET processed=%s, processed_at=NOW(), error_message=%s
WHERE load_id=%s AND uei_sam=%s
```

## Replay (Follow-Up — Not Blocking This Phase)

Once staging is populated, a `--from-staging` flag on `load` commands enables re-normalization without API calls:

```bash
python main.py load awards --from-staging --load-id 42   # replay specific load
python main.py load awards --from-staging                # replay all unprocessed
```

Implementation: add `load_from_staging(load_id=None)` to each loader that SELECTs `raw_json` from staging and calls the existing `load_*` method with that data. Mark staging records `processed='Y'` after re-processing.

**This is not implemented in this phase** — staging write is the value, replay is a follow-up.
