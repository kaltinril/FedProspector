# Phase 44.11: Entity Search Performance

**Status**: PLANNED
**Priority**: High — user-facing search is noticeably slow
**Discovered**: Manual testing, searching for "msone"

## Problem Statement

Entity search on the UI takes multiple seconds. The C# `EntityService.SearchAsync()` uses `LIKE '%search%'` (leading wildcard) against 700K+ rows in the `entity` table. A leading-wildcard LIKE cannot use a B-tree index, so MySQL does a full table scan. The scan runs **twice** per search — once for COUNT, once for paginated results.

## Root Cause

- `EntityService.cs` builds `e.LegalBusinessName.Contains(name)` which EF Core translates to `LIKE '%msone%'`
- `entity` table has `INDEX idx_entity_name (legal_business_name)` — B-tree only, useless for leading-wildcard
- `SearchAsync` calls `CountAsync()` then a separate paginated query — double scan

## Fixes

### 1. Add FULLTEXT index on `entity.legal_business_name`

```sql
ALTER TABLE entity ADD FULLTEXT INDEX ft_entity_name (legal_business_name);
```

DDL file: `fed_prospector/db/schema/tables/20_entity.sql`

### 2. Switch C# service to `MATCH ... AGAINST`

In `EntityService.cs`, replace the LIKE filter with raw SQL `MATCH(legal_business_name) AGAINST(@term IN BOOLEAN MODE)` when search term is 3+ characters. Fall back to LIKE for 1-2 character terms.

### 3. Eliminate double scan

Replace separate `CountAsync()` + paginated query with a single query using `COUNT(*) OVER()` window function to get total count alongside results.

## Files to Change

- `fed_prospector/db/schema/tables/20_entity.sql` — Add FULLTEXT index to DDL
- `api/src/FedProspector.Infrastructure/Services/EntityService.cs` — MATCH AGAINST + single-query pagination
- Live DB — `ALTER TABLE entity ADD FULLTEXT INDEX ft_entity_name (legal_business_name)`

## Verification

- Entity search for "msone" returns results in < 1 second
- Pagination total count is correct
- Short search terms (1-2 chars) still work (LIKE fallback)
