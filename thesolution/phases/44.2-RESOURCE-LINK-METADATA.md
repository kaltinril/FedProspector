# Phase 44.2: Resource Link Metadata Enrichment

**Status**: PLANNED
**Depends on**: Phase 41 (Detail Views)

## Context

The Opportunity Detail page shows "Resource Links" as raw SAM.gov API URLs (e.g., `https://sam.gov/api/prod/opps/v3/opportunities/resources/files/4346cbda.../download`). SAM.gov's public API only returns URL strings -- no filenames, descriptions, or metadata. Users need friendly filenames to understand what each attachment is before clicking.

## Approach: Lazy-Resolve with DB Caching

When a user views an opportunity with unresolved resource links, the UI auto-fires a backend resolve call. The C# API makes HEAD requests to SAM.gov URLs, extracts filename/content-type from HTTP headers, stores metadata in a new DB table, returns it. Subsequent visits show cached filenames instantly.

## Implementation Steps

### Step 1: Database -- New `opportunity_resource_link` table

**New file**: `fed_prospector/db/schema/tables/31_opportunity_resource_link.sql`

```sql
CREATE TABLE IF NOT EXISTS opportunity_resource_link (
    id               INT AUTO_INCREMENT PRIMARY KEY,
    notice_id        VARCHAR(100) NOT NULL,
    url              VARCHAR(1000) NOT NULL,
    url_index        SMALLINT NOT NULL,
    filename         VARCHAR(500),
    content_type     VARCHAR(200),
    file_size_bytes  BIGINT,
    resolved_at      DATETIME,
    resolve_error    VARCHAR(500),
    UNIQUE KEY uq_orl_notice_url (notice_id, url(255)),
    INDEX idx_orl_notice (notice_id),
    CONSTRAINT fk_orl_notice FOREIGN KEY (notice_id) REFERENCES opportunity(notice_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

Separate table preserves ETL schema ownership -- the ETL-owned `resource_links` JSON column is not modified.

**Modify**: `api/src/FedProspector.Infrastructure/Services/OpportunityService.cs`
- `GetDetailAsync`: Join with `opportunity_resource_link` to return cached metadata; URLs without cached rows return `IsResolved = false`
- New `ResolveResourceLinksAsync`: HEAD-request each unresolved URL, parse headers, upsert rows

Header parsing priority:
1. `Content-Disposition: attachment; filename="name.pdf"` -- extract filename
2. Fallback: URL-decode last path segment before `/download`
3. `Content-Type` -- store as-is for file icon
4. `Content-Length` -- store for file size display
5. On failure: store error in `resolve_error`, return null filename

## UX Flow

1. User opens opportunity detail -- API returns resource links with cached metadata (if any)
2. Unresolved links show as "Attachment 1", "Attachment 2"
3. Component auto-fires resolve on mount if any unresolved
4. Backend HEAD-requests SAM.gov, caches results, returns metadata
5. UI refreshes to show filenames (e.g., "Performance_Work_Statement.pdf (2.4 MB)")
6. Future visits: all metadata from cache, no HEAD requests

## Error Handling

| Scenario | Behavior |
|----------|----------|
| HEAD returns 403/401 | Store error, show "Attachment N" |
| HEAD times out | Store error, show "Attachment N" |
| HEAD returns 404 | Store error, show "Attachment N" with warning |
| No Content-Disposition | Try URL-based filename extraction |
| Malformed URL | Store error, show raw URL as fallback |
