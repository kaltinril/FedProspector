# Phase 44.2: Resource Link Metadata Enrichment

**Status**: COMPLETE
**Depends on**: Phase 41 (Detail Views)

## Context

The Opportunity Detail page shows "Resource Links" as raw SAM.gov API URLs (e.g., `https://sam.gov/api/prod/opps/v3/opportunities/resources/files/4346cbda.../download`). SAM.gov's public API only returns URL strings -- no filenames, descriptions, or metadata. Users need friendly filenames to understand what each attachment is before clicking.

## Pre-Implementation Validation -- COMPLETED 2026-03-06

### 1. SAM.gov HEAD Request Support -- CONFIRMED WORKING

Tested HEAD requests against real SAM.gov resource link URLs. Results:

- **HTTP response**: `303 See Other` (redirect to S3 pre-signed URL)
- **Content-Disposition**: YES -- `attachment; filename="P09 SOW Disinterment Support Svc 01.05.26.docx"`
- **Content-Type**: YES -- `application/vnd.openxmlformats-officedocument.wordprocessingml.document`
- **Content-Length**: Not on initial 303, but available on the S3 redirect target
- **Location header**: Redirect to `iae-fbo-attachments.s3.amazonaws.com` with signed URL (9-second expiry)

Key finding: The 303 response itself contains both `Content-Disposition` (with filename) and `Content-Type`. Do NOT follow the redirect -- that would download the actual file. Read headers from the 303 directly.

Tested on 2 different URLs -- behavior is consistent. ~30ms per request. No rate limiting. No API key required.

### 2. URL Length Analysis

Queried all 21,843 resource link URLs across 4,585 opportunities:

| Metric | Value |
|--------|-------|
| Min URL length | 104 chars |
| Max URL length | 104 chars |
| Avg URL length | 104 chars |
| URLs > 255 chars | 0 |

All URLs follow the exact same pattern: `https://sam.gov/api/prod/opps/v3/opportunities/resources/files/{32-char-hex}/download` = 104 characters.

### 3. Decision: ETL-Time Enrichment (not Lazy-Resolve)

Originally planned as lazy-resolve (C# API makes HEAD requests when users view opportunities). Validation showed HEAD requests are fast (~30ms), free (no rate limit, no API key), and reliable. This eliminates the justification for lazy-resolve complexity.

**New approach: Enrich the `resource_links` JSON during Python ETL loading.** This is dramatically simpler:
- No new database table
- No new C# API endpoint, HttpClient, or resolve service
- No frontend resolve mutation, retry logic, or status tracking
- Data is ready before any user views it -- no "Attachment 1" flicker
- Fits the existing ETL pattern: Python gathers data, C# serves it
- 21,843 URLs at ~30ms each = ~11 minutes for full backfill

## Approach: ETL-Time Enrichment

During opportunity loading, the Python ETL enriches each resource link URL with filename and content-type by making a HEAD request to SAM.gov. The existing `resource_links` JSON column is updated from a simple URL array to an array of objects with metadata. A standalone CLI command handles backfilling existing data.

## Implementation Steps

### Step 1: Python -- Resource Link Resolver Utility

**New file**: `fed_prospector/etl/resource_link_resolver.py`

Utility function that takes a list of resource link URLs and returns enriched metadata:

```python
def resolve_resource_links(urls: list[str], max_concurrent: int = 5) -> list[dict]:
    """HEAD-request each URL, extract filename/content-type from 303 response."""
```

For each URL:
1. Validate URL starts with `https://sam.gov/` or `https://api.sam.gov/` (SSRF protection)
2. Send HEAD request with `allow_redirects=False`
3. On 303: parse `Content-Disposition` header for filename, extract `Content-Type`
4. Fallback: URL-decode last path segment if no `Content-Disposition`
5. On error (timeout, 404, 403): log warning, return URL with `filename: null`

Returns list of objects: `{"url": "...", "filename": "...", "content_type": "..."}`

Use `asyncio` or `concurrent.futures.ThreadPoolExecutor` with max 5 workers for parallelism.

### Step 2: Integrate into Opportunity Loader

**Modify**: `fed_prospector/etl/opportunity_loader.py`

After loading opportunities into the database, for any opportunity with `resource_links` containing plain URL strings (not yet enriched objects), call `resolve_resource_links()` and UPDATE the `resource_links` JSON column with the enriched version.

Detection: if `resource_links` JSON array contains strings (old format), enrich. If it contains objects with `filename` key (new format), skip.

### Step 3: CLI Command for Backfill

**Modify**: `fed_prospector/main.py` (CLI)

Add command: `python main.py update link-metadata [--missing-only] [--batch-size 100]`

- Queries opportunities where `resource_links` contains plain URL strings (not enriched)
- HEAD-requests each URL, enriches the JSON, updates the row
- `--missing-only` (default): only process un-enriched links
- `--batch-size`: process N opportunities per batch, commit between batches
- Progress logging: "Enriched 500/4585 opportunities (2,341 links resolved)"

### Step 4: C# API -- Parse Enriched JSON

**Modify**: `api/src/FedProspector.Infrastructure/Services/OpportunityService.cs`

In `GetDetailAsync`, change how `ResourceLinks` is returned. Currently passes raw JSON string through. Now parse the JSON and return structured data.

**Modify**: `api/src/FedProspector.Core/DTOs/Opportunities/OpportunityDetailDto.cs`
- Keep `ResourceLinks` as `string?` for backward compatibility
- Add NEW property `ResourceLinkDetails: List<ResourceLinkDto>` populated by parsing the enriched JSON

**New**: `api/src/FedProspector.Core/DTOs/Opportunities/ResourceLinkDto.cs`

### Step 5: React Frontend -- Display Filenames

## JSON Format Migration

### Before (current)
```json
["https://sam.gov/api/.../download", "https://sam.gov/api/.../download"]
```

### After (enriched)
```json
[
  {"url": "https://sam.gov/api/.../download", "filename": "SOW_Document.pdf", "content_type": "application/pdf"},
  {"url": "https://sam.gov/api/.../download", "filename": null, "content_type": null}
]
```

The C# API and React UI handle both formats during migration. Once backfill is complete, the old string-only format no longer exists.

## Error Handling

| Scenario | Behavior |
|----------|----------|
| HEAD returns 303 with Content-Disposition | Extract filename and content-type (happy path) |
| HEAD returns 403/401/404 | Log warning, store URL with `filename: null` |
| HEAD times out | Log warning, store URL with `filename: null` |
| Non-SAM.gov URL | Skip (SSRF protection), store URL with `filename: null` |
| Malformed URL | Skip, store as-is |

Failed URLs display as "Attachment N" in the UI -- same as today. No retry complexity needed; re-run `update link-metadata` to retry failures.
