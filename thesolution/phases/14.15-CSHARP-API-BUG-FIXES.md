# Phase 14.15 — C# API Bug Fixes

## Status: COMPLETE (2026-03-03)

## Goal

Code review of the ASP.NET Core API found critical security, data integrity, and performance issues. This phase fixes them in priority order. The C# API is in `api/src/FedProspector.Api/` and `api/src/FedProspector.Infrastructure/`.

## CRITICAL Fixes

### Fix 1 — Admin Privilege Escalation: Any Admin Can Create Organizations

**File:** `api/src/FedProspector.Api/Controllers/AdminController.cs:73-88`

**Problem:** `CreateOrganization` and `CreateOwner` are documented as "System Admin only" but the controller-level `[Authorize(Roles = "admin")]` applies equally to all endpoints. There is no policy or secondary check distinguishing a system admin from an org-level admin. Any user with the `admin` role can create new tenants — a highly privileged operation.

**Fix:** Add an `is_system_admin` flag to `AppUser` (or use a dedicated role/claim) and add a check at the top of both methods:
```csharp
// In CreateOrganization and CreateOwner:
var isSystemAdmin = User.HasClaim("is_system_admin", "true");
if (!isSystemAdmin)
    return Forbid();
```
Alternatively, introduce a `"system_admin"` role distinct from `"admin"` and add `[Authorize(Roles = "system_admin")]` to those two action methods only. Coordinate with the auth service to assign this role at user creation time.

### Fix 2 — Refresh Token Stored in Semantically Wrong Column

**File:** `api/src/FedProspector.Infrastructure/Services/AuthService.cs:148-150, 432-452`

**Problem:** The refresh token hash is stored in `AppSession.RevokedReason` — a field meant to record why a session was revoked. The comment acknowledges this: `"Store refresh token hash in RevokedReason field temporarily (repurposed as refresh_token_hash..."`. This causes: (1) refresh token lookup collides with real revocation reasons; (2) token-reuse detection can false-trigger and lock out innocent users; (3) audit logs reading `RevokedReason` get garbage data for active sessions.

**Fix:** Add a dedicated `refresh_token_hash` column to `app_session` via an EF Core migration:
```csharp
// New migration:
migrationBuilder.AddColumn<string>(
    name: "refresh_token_hash",
    table: "app_session",
    type: "varchar(64)",
    nullable: true);
migrationBuilder.CreateIndex(
    name: "IX_app_session_refresh_token_hash",
    table: "app_session",
    column: "refresh_token_hash");
```
Update `AuthService` to store the hash in `RefreshTokenHash` and query on it. Remove the `RevokedReason` dual-use. Clear `RefreshTokenHash` when a session is revoked.

### Fix 3 — `ChangePassword` Exceptions Not Caught; Session Cleared Before Confirmation

**File:** `api/src/FedProspector.Api/Controllers/AuthController.cs:154-168`

**Problem:** `AuthController.ChangePassword` calls `_authService.ChangePasswordAsync(...)` without catching exceptions. If the service throws `InvalidOperationException` ("Current password is incorrect"), the middleware converts it to a generic 400 — losing the specific error text. Worse, on the success path the controller clears auth cookies (line 165) before awaiting completion, so if the async call fails after the cookies are cleared, the user is logged out without their password being changed.

**Fix:**
```csharp
try
{
    await _authService.ChangePasswordAsync(userId.Value, request.CurrentPassword, request.NewPassword);
    ClearAuthCookies();
    return Ok(new { message = "Password changed successfully. Please log in again." });
}
catch (InvalidOperationException ex)
{
    return BadRequest(new { message = ex.Message });
}
catch (KeyNotFoundException)
{
    return NotFound(new { message = "User not found." });
}
```
Ensure `ClearAuthCookies()` is only called after the service confirms success.

### Fix 4 — Cross-Tenant Prospect Data Leak in Opportunity Search

**File:** `api/src/FedProspector.Infrastructure/Services/OpportunityService.cs:59-89`

**Problem:** The opportunity search joins to `_context.Prospects` without filtering by `organizationId`. If two organizations prospect the same `notice_id`, both orgs' prospect status and assigned user appear in search results for any user who queries that opportunity. An Org A user can see that Org B has a prospect on opportunity XYZ and who it's assigned to.

**Fix:** Add the org filter to the prospect join:
```csharp
.Join(
    _context.Prospects.Where(p => p.OrganizationId == organizationId),
    o => o.NoticeId,
    p => p.NoticeId,
    (o, p) => new { Opportunity = o, Prospect = p }
)
```
Or use a left join that filters by org:
```csharp
from o in query
join p in _context.Prospects.Where(p => p.OrganizationId == organizationId)
    on o.NoticeId equals p.NoticeId into prospects
from p in prospects.DefaultIfEmpty()
select new { ... }
```

### Fix 5 — Temporary Password Returned in Plain Text in API Response

**File:** `api/src/FedProspector.Infrastructure/Services/AdminService.cs:186-190`

**Problem:** `ResetPasswordAsync` returns `TemporaryPassword` in the JSON response body. This plaintext password appears in application logs, browser dev tools, proxy logs, and any API gateway access logs. If log storage is compromised, all admin-reset passwords are exposed.

**Fix:** Do not return the temporary password in the response. Instead, send it to the user's registered email address (requires email service integration). For now, if email is not yet implemented, return only a confirmation message and log the temp password internally at DEBUG level (not INFO/ERROR):
```csharp
_logger.LogDebug("Temporary password for user {UserId}: {TempPassword}", userId, tempPassword);
return new ResetPasswordResponse
{
    Message = $"Password reset for user '{user.Username}'. Temporary credentials sent via email."
};
```
Track email delivery as a follow-up task (Phase 15 or later).

### Fix 6 — `CaptureManagerId` Not Validated Against Organization

**File:** `api/src/FedProspector.Infrastructure/Services/ProspectService.cs:58-65`

**Problem:** `CreateAsync` validates `AssignedTo` against the organization but does NOT validate `CaptureManagerId`. An authenticated user from Org A can supply a `CaptureManagerId` belonging to a user in Org B. That cross-org user ID is stored and returned in detail views, leaking that the user ID exists.

**Fix:** Add the same org-membership validation for `CaptureManagerId`:
```csharp
if (request.CaptureManagerId.HasValue)
{
    var captureManager = await _db.AppUsers
        .FirstOrDefaultAsync(u => u.UserId == request.CaptureManagerId.Value
                                  && u.OrganizationId == organizationId);
    if (captureManager == null)
        throw new KeyNotFoundException("CaptureManagerId not found in organization.");
}
```

### Fix 7 — CSV Injection in Opportunity Export

**File:** `api/src/FedProspector.Infrastructure/Services/OpportunityService.cs:368`

**Problem:** Only 5 string fields are quoted/escaped in the CSV output. `SetAsideCode`, `NaicsCode`, `PopState`, `AwardAmount`, and `Active` are written unquoted. Values beginning with `=`, `+`, `-`, or `@` are interpreted as formulas by Excel/LibreOffice Calc — a CSV injection vector.

**Fix:** Quote and escape all fields:
```csharp
private static string CsvField(object? value)
{
    var s = value?.ToString() ?? "";
    // Neutralize formula injection
    if (s.Length > 0 && "=+-@".Contains(s[0]))
        s = "'" + s;
    return $"\"{s.Replace("\"", "\"\"")}\"";
}

// Replace the AppendLine with:
sb.AppendLine($"{CsvField(item.NoticeId)},{CsvField(item.Title)},{CsvField(item.SolicitationNumber)},{CsvField(item.DepartmentName)},{CsvField(item.Office)},{CsvField(item.PostedDate)},{CsvField(item.ResponseDeadline)},{CsvField(item.SetAsideCode)},{CsvField(item.NaicsCode)},{CsvField(item.PopState)},{CsvField(item.AwardAmount)},{CsvField(item.Active)}");
```

### Fix 8 — `FilterCriteria` Stored as Raw String Without Validation

**File:** `api/src/FedProspector.Infrastructure/Services/SavedSearchService.cs:195`; `api/src/FedProspector.Core/DTOs/SavedSearches/UpdateSavedSearchRequest.cs:7`

**Problem:** `UpdateSavedSearchRequest.FilterCriteria` is `string?` (raw JSON string) while `CreateSavedSearchRequest.FilterCriteria` is a typed `SavedSearchFilterCriteria` object. On update, the raw string is stored without JSON validation. When `RunAsync` later deserializes it, malformed JSON silently returns null (no results, no error surfaced to user).

**Fix:** Change `UpdateSavedSearchRequest.FilterCriteria` from `string?` to `SavedSearchFilterCriteria?` (same type as create). Update the service to serialize it properly:
```csharp
if (request.FilterCriteria != null)
    search.FilterCriteria = JsonSerializer.Serialize(request.FilterCriteria);
```

### Fix 9 — N+1 Queries: `ProposalService.ListAsync` (75+ DB Calls Per Page)

**File:** `api/src/FedProspector.Infrastructure/Services/ProposalService.cs:379-416`

**Problem:** `ListAsync` paginates proposals, then calls `BuildDetailAsync` in a loop for each result. `BuildDetailAsync` executes 3 separate DB queries per proposal. For a page of 25 proposals: 25 × 3 = 75+ round-trips.

**Fix:** Load the page of proposals with all needed data in a single query using `.Include()`:
```csharp
var proposals = await _db.Proposals
    .Include(p => p.Prospect)
        .ThenInclude(pr => pr.Opportunity)
    .Include(p => p.Milestones)
    .Include(p => p.Documents)
    .Where(p => p.Prospect.OrganizationId == organizationId)
    .OrderByDescending(p => p.UpdatedAt)
    .Skip((request.Page - 1) * request.PageSize)
    .Take(request.PageSize)
    .ToListAsync();
```
Map directly to `ProposalListItemDto` without calling `BuildDetailAsync` per item.

### Fix 10 — N+1 Queries: `ProspectService.GetDetailAsync` Called After Every Write

**File:** `api/src/FedProspector.Infrastructure/Services/ProspectService.cs:234-363`

**Problem:** `GetDetailAsync` issues 7+ sequential DB queries and is called as the return value of every write operation (UpdateStatus, Reassign, AddNote, AddTeamMember, RecalculateScore). Every mutation triggers a 7-query read.

**Fix:** Consolidate the 7 queries into one with eager loading:
```csharp
var prospect = await _db.Prospects
    .Include(p => p.Opportunity)
    .Include(p => p.AssignedUser)
    .Include(p => p.CaptureManager)
    .Include(p => p.Notes.OrderByDescending(n => n.CreatedAt))
    .Include(p => p.TeamMembers).ThenInclude(tm => tm.User)
    .Include(p => p.Proposal)
    .FirstOrDefaultAsync(p => p.ProspectId == id && p.OrganizationId == organizationId);
```

## MEDIUM Fixes

### Fix 11 — Missing Validators for `CreateOrganizationRequest` and `CreateOwnerRequest`

**File:** `api/src/FedProspector.Core/Validators/`

**Problem:** No validators exist for either request. `CreateOrganizationRequest` has no length limit on `Name` or `Slug`, no slug format validation (alphanumeric + hyphens), and no non-empty requirement. `CreateOwnerRequest` has no password policy (minimum length, complexity).

**Fix:** Create `CreateOrganizationRequestValidator`:
```csharp
RuleFor(x => x.Name).NotEmpty().MaximumLength(200);
RuleFor(x => x.Slug).NotEmpty().MaximumLength(100).Matches("^[a-z0-9-]+$")
    .WithMessage("Slug must contain only lowercase letters, numbers, and hyphens.");
```

Create `CreateOwnerRequestValidator`:
```csharp
RuleFor(x => x.Username).NotEmpty().MaximumLength(50);
RuleFor(x => x.Email).NotEmpty().EmailAddress();
RuleFor(x => x.Password).NotEmpty().MinimumLength(8).MaximumLength(100);
```

### Fix 13 — `Logout` Returns 404 After Cookies Already Cleared

**File:** `api/src/FedProspector.Api/Controllers/AuthController.cs:59-89`

**Problem:** `Logout` clears auth cookies unconditionally, then if no session is found, returns `NotFound`. The user is effectively logged out but receives a 404 response, confusing well-behaved clients that check status codes.

**Fix:** Return 200 regardless of whether a session was found in the database. The cookies are cleared — the client is logged out. Session not found is not an error from the client's perspective:
```csharp
ClearAuthCookies();
await _authService.LogoutAsync(sessionToken);  // fire and forget if not found
return Ok(new { message = "Logged out successfully." });
```

### Fix 14 — `StalenessThresholds` Duplicated Between Two Files

**Files:** `api/src/FedProspector.Infrastructure/Services/AdminService.cs:17-27`; `api/src/FedProspector.Api/Controllers/HealthController.cs:19-29`

**Problem:** Identical `StalenessThresholds` dictionaries in two places. If a threshold is updated in one, the other diverges silently, giving contradictory freshness verdicts between the health and admin endpoints.

**Fix:** Extract to a shared constant in `api/src/FedProspector.Core/`:
```csharp
// FedProspector.Core/Constants/EtlStalenessThresholds.cs
public static class EtlStalenessThresholds
{
    public static readonly Dictionary<string, int> Hours = new()
    {
        ["SAM_OPPORTUNITY"] = 24,
        ["SAM_ENTITY"] = 168,
        // ... etc
    };
}
```
Both `AdminService` and `HealthController` reference `EtlStalenessThresholds.Hours`.

### Fix 16 — `PUT` Method Missing from CSRF Mutation Check

**File:** `api/src/FedProspector.Api/Middleware/CsrfMiddleware.cs:16-19`

**Problem:** `MutatingMethods` set contains `POST`, `PATCH`, `DELETE` but not `PUT`. Future endpoints using `PUT` will bypass CSRF protection silently.

**Fix:**
```csharp
private static readonly HashSet<string> MutatingMethods = new(StringComparer.OrdinalIgnoreCase)
{
    "POST", "PUT", "PATCH", "DELETE"
};
```
