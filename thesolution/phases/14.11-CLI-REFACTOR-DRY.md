## The Five Wins

### Win 1 — `print_table`: Eliminate ~150 Lines of Duplicated ASCII Table Formatting

**The problem:** Every search command manually formats output into an ASCII table. Each one re-implements: print the header row, print the separator, loop rows, truncate values, print each cell. This is 20–40 lines per command, repeated in 7 files:

```
awards.py, entities.py, opportunities.py, subaward.py,
fedhier.py, prospecting.py, health.py
```

Typical pattern (appears in every file):
```python
click.echo(f"{'PIID':<17} {'VENDOR':<30} {'AGENCY':<25} ...")
click.echo("-" * 100)
for row in rows:
    piid = str(row["piid"] or "")[:17]
    vendor = str(row["vendor_name"] or "")[:30]
    click.echo(f"{piid:<17} {vendor:<30} ...")
```

**The fix:** Add `print_table(columns, rows)` to `fed_prospector/cli/cli_utils.py`:

```python
def print_table(columns: list[dict], rows: list) -> None:
    """
    Print a fixed-width ASCII table to stdout.

    columns: list of {"header": str, "width": int}
             e.g. [{"header": "PIID", "width": 17}, {"header": "VENDOR", "width": 30}]
    rows:    list of sequences (tuple, list) or dicts keyed by column position.
             Values are coerced to str, truncated to column width, left-justified.
    """
    headers = [col["header"] for col in columns]
    widths  = [col["width"]  for col in columns]

    header_line = " ".join(h.ljust(w) for h, w in zip(headers, widths))
    separator   = "-" * len(header_line)

    click.echo(header_line)
    click.echo(separator)
    for row in rows:
        vals = row if not isinstance(row, dict) else row.values()
        cells = [str(v or "")[:w].ljust(w) for v, w in zip(vals, widths)]
        click.echo(" ".join(cells))
```

**Per-file changes:** Remove the inline header/separator/loop block; replace with one `print_table(columns, rows)` call. The `columns` list is defined once per command as a module-level or local constant.

**Estimated savings:** ~150 lines across 7 files.

### Win 2 — `QueryBuilder`: Eliminate ~100 Lines of Duplicated WHERE Clause Assembly

**The problem:** All search commands build SQL WHERE clauses with the same boilerplate pattern. It appears in 8 files:

```
awards.py, entities.py, opportunities.py, subaward.py,
fedhier.py, exclusions.py, prospecting.py, spending.py
```

Exact pattern in every file:
```python
where_clauses = []
params = []
if naics_code:
    where_clauses.append("naics_code = %s")
    params.append(naics_code)
if agency_name:
    where_clauses.append("agency_name LIKE %s")
    params.append(f"%{agency_name}%")
where_sql = ("WHERE " + " AND ".join(where_clauses)) if where_clauses else ""
```

**The fix:** Add a `QueryBuilder` class to `fed_prospector/cli/cli_utils.py`:

```python
class QueryBuilder:
    """
    Accumulates SQL WHERE conditions and parameter values.

    Usage:
        qb = QueryBuilder()
        qb.filter("naics_code = %s", naics_code)
        qb.filter("agency_name LIKE %s", f"%{agency}%")
        where_sql, params = qb.build_where()
        cursor.execute(f"SELECT ... FROM ... {where_sql} LIMIT %s", params + [limit])
    """

    def __init__(self):
        self._clauses: list[str] = []
        self._params:  list     = []

    def filter(self, condition: str, *values) -> "QueryBuilder":
        """Add a condition only when all its values are truthy. Returns self for chaining."""
        if all(v is not None and v != "" for v in values):
            self._clauses.append(condition)
            self._params.extend(values)
        return self

    def build_where(self) -> tuple[str, list]:
        """Return (where_sql, params). where_sql is '' when no filters are active."""
        if not self._clauses:
            return "", []
        return "WHERE " + " AND ".join(self._clauses), list(self._params)
```

**Per-file changes:** Remove the local `where_clauses`/`params` setup block; replace with `QueryBuilder` calls. The `filter()` method skips None/empty values automatically, so existing `if field:` guards can be removed.

**Estimated savings:** ~100 lines across 8 files.

### Win 3 — `get_cursor` Context Manager: Eliminate ~80 Lines of try/finally in CLI

**The problem:** Phase 14.10 adds `get_cursor(dictionary=False)` to `fed_prospector/db/connection.py`. The CLI has 11+ commands that still manually open/close connections with 8-line try/finally blocks. Those should adopt the context manager.

Current pattern in every read-only CLI command:
```python
conn = get_connection()
cursor = conn.cursor(dictionary=True)
try:
    cursor.execute(sql, params)
    rows = cursor.fetchall()
finally:
    cursor.close()
    conn.close()
```

**The fix** (context manager is already defined in Phase 14.10):
```python
with get_cursor(dictionary=True) as cursor:
    cursor.execute(sql, params)
    rows = cursor.fetchall()
```

**Scope:** Read-only query commands only. Commands that need to control commit timing (write operations with explicit `conn.commit()`) keep explicit connection management.

**Affected files:** awards.py, entities.py, opportunities.py, subaward.py, fedhier.py, exclusions.py, prospecting.py, spending.py, health.py, admin.py (read queries only).

**Estimated savings:** ~80 lines across 11 CLI files.

### Win 4 — Entity Load Helper Extraction: Eliminate ~140 Lines of Intra-File Duplication

**The problem:** In `entities.py`, the `load_entities` and `refresh_entities` commands are independent Click commands that share no code — but both contain identical logic for processing `.dat` files (~70 lines) and identical logic for processing `.json` files (~70 lines). This is duplication within a single file.

```
load_entities()     → dat block (70 lines) + json block (70 lines)
refresh_entities()  → dat block (70 lines) + json block (70 lines)
```

**The fix:** Extract two private helper functions in `entities.py`:

```python
def _load_dat_file(file_path: str, loader, load_manager) -> dict:
    """Process a single .dat file through the entity loader. Returns stats dict."""
    ...

def _load_json_file(file_path: str, loader, load_manager) -> dict:
    """Process a single .json file through the entity loader. Returns stats dict."""
    ...
```

Both `load_entities` and `refresh_entities` call these helpers instead of repeating the blocks inline.

**Estimated savings:** ~140 lines (the duplicate blocks are removed; the helpers add ~10 lines of function signature overhead).

### Win 5 — `echo_load_stats`: Eliminate ~40 Lines of Repeated Stats Output

**The problem:** All 9 load commands print the same 5-line stats block after loading:

```python
click.echo(f"  Records read:      {stats['records_read']:>10,d}")
click.echo(f"  Records inserted:  {stats['records_inserted']:>10,d}")
click.echo(f"  Records updated:   {stats['records_updated']:>10,d}")
click.echo(f"  Records skipped:   {stats['records_skipped']:>10,d}")
click.echo(f"  Records errored:   {stats['records_errored']:>10,d}")
```

5 lines × 9 commands = 45 lines of identical code.

**The fix:** Add `echo_load_stats(stats)` to `fed_prospector/cli/cli_utils.py`:

```python
def echo_load_stats(stats: dict) -> None:
    """Print the standard ETL load stats block to stdout."""
    click.echo(f"  Records read:      {stats.get('records_read', 0):>10,d}")
    click.echo(f"  Records inserted:  {stats.get('records_inserted', 0):>10,d}")
    click.echo(f"  Records updated:   {stats.get('records_updated', 0):>10,d}")
    click.echo(f"  Records skipped:   {stats.get('records_skipped', 0):>10,d}")
    click.echo(f"  Records errored:   {stats.get('records_errored', 0):>10,d}")
```

**Per-file changes:** Replace the 5-line inline block with `echo_load_stats(stats)`.

**Estimated savings:** ~40 lines across 9 files (awards.py, entities.py, opportunities.py, subaward.py, fedhier.py, exclusions.py, spending.py, health.py, admin.py).
