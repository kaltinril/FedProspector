# Phase 14.26: Resumable Paginated Loads

**Status**: PARTIAL (entities done, opportunities + fed hierarchy pending)

## Overview

Paginated vendor API loads (entities, opportunities, fed hierarchy) can take many API calls.
If the process dies mid-run (kill, Ctrl+C, network error), all progress is lost and API budget
is wasted on re-fetching pages already processed. This phase adds per-page checkpointing and
resume-from-last-page support.

## What Was Implemented: Entity Daily Refresh

### Behavior

- `entities-refresh --type=daily` now processes page-by-page instead of collecting all pages
  then bulk-loading
- After each page: entities are upserted into the database, then `etl_load_log` is updated
  with `pages_fetched`, `total_records`, and `complete` flag in the `parameters` JSON column
- **Resume**: on restart, queries `etl_load_log` for the highest `pages_fetched` value for
  this date (any status), and resumes from that page
- **KeyboardInterrupt handling**: catches Ctrl+C and chained MySQL `InternalError` (from
  interrupted connections), does not mark the load as FAILED
- **CLI options**: `--max-calls` (API budget control), `--force` (ignore previous progress
  and start fresh), `--key` (API key selection 1 or 2)

### Entity Management API Details

- Pagination style: page/size (not offset)
- 10 records per page
- 0-based page numbering

## Pattern for Applying to Other Loaders

The resumable pagination pattern has four parts:

### 1. Client: Add Page Iterator

Add an `iter_<thing>_pages()` method that yields `(results_list, page_num_or_offset, total_records)`
per API call. This is a generator that handles pagination internally and yields one tuple per page.

### 2. Loader: Add Batch Method

Add a `load_<thing>_batch(items, load_id)` method that processes items under an existing `load_id`
without creating or completing the load entry. The CLI manages the load lifecycle.

### 3. CLI: Manage Load Lifecycle

The CLI command directly manages the load entry:
- Create load entry at start (or find existing one for resume)
- Iterate pages from the client generator
- Upsert each page's records via the loader batch method
- Save progress after each page via `save_load_progress()`
- Handle KeyboardInterrupt gracefully (don't mark as FAILED)
- Mark complete only when all pages are processed

### 4. Resume Query

Query `etl_load_log` for the highest checkpoint for the current run:

```sql
SELECT load_id, parameters
FROM etl_load_log
WHERE source = '<source_name>'
  AND parameters LIKE '%"date_key": "YYYY-MM-DD"%'
ORDER BY CAST(JSON_EXTRACT(parameters, '$.pages_fetched') AS UNSIGNED) DESC
LIMIT 1
```
