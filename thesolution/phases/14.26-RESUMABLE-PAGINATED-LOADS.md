# Phase 14.26: Resumable Paginated Loads

**Status**: COMPLETE

## Overview

Paginated vendor API loads (entities, opportunities, fed hierarchy) can take many API calls.
If the process dies mid-run (kill, Ctrl+C, network error), all progress is lost and API budget
is wasted on re-fetching pages already processed. This phase adds per-page checkpointing and
resume-from-last-page support.

## What Was Implemented

### 1. Entity Daily Refresh (first implementation)

- `entities-refresh --type=daily` processes page-by-page instead of collecting all pages then bulk-loading
- After each page: entities are upserted, then `etl_load_log` is updated with `pages_fetched`, `total_records`, and `complete` flag in `parameters` JSON
- **Resume**: queries `etl_load_log` for highest `pages_fetched` for this date, resumes from that page
- **KeyboardInterrupt handling**: catches Ctrl+C and chained MySQL `InternalError`, does not mark FAILED
- **CLI options**: `--max-calls`, `--force`, `--key`
- Entity Management API: page/size pagination, 10 records/page, 0-based

### 2. Opportunity Loader

- `load-opportunities` processes page-by-page with full resume support
- **Single set-aside path**: straightforward page iterator, saves progress after each page
- **Multi-set-aside path**: processes each set-aside sequentially; tracks `completed_set_asides`, `current_set_aside`, `current_pages_in_set_aside` in parameters JSON; on resume skips completed set-asides and resumes current one from saved page
- **Dedup**: handled by DB upsert (`INSERT ... ON DUPLICATE KEY UPDATE` on `notice_id`) — no in-memory dict needed
- **Date-range chunking**: handled inside `iter_opportunity_pages()`; single-chunk uses `offset_start` on `paginate()` to skip already-loaded pages efficiently; multi-chunk accepts re-fetch cost (rare historical loads only)
- **CLI options**: `--force` (new), existing `--max-calls`, `--set-aside`, `--days-back`, `--posted-from`/`--posted-to`, `--historical`
- Opportunity API: offset pagination, 1000 records/page

### 3. Fed Hierarchy Loader

- `load-hierarchy` (incremental path) processes page-by-page with full resume support
- Resume keyed on `status_filter` (default "Active") in parameters JSON
- `--full-refresh` path remains non-resumable (DELETEs all records first)
- **CLI options**: `--force` (new), existing `--max-calls`, `--status`
- Fed Hierarchy API: offset pagination, 100 records/page

## Resumable Pattern (Reference)

The pattern has four parts:

### 1. Client: Page Iterator
`iter_<thing>_pages()` yields `(results_list, page_num_or_offset, total_records)` per API call.

### 2. Loader: Batch Method
`load_<thing>_batch(items, load_id)` processes items under an existing `load_id` without managing load lifecycle.

### 3. CLI: Load Lifecycle
- Create load entry at start (or find existing one for resume)
- Iterate pages, upsert each page's records, save progress after each page
- Handle KeyboardInterrupt gracefully (don't mark FAILED)
- Mark complete only when all pages processed

### 4. Resume Query
```sql
SELECT load_id, parameters FROM etl_load_log
WHERE source_system = '<source>' AND parameters LIKE '%"key": "value"%'
ORDER BY CAST(JSON_EXTRACT(parameters, '$.pages_fetched') AS UNSIGNED) DESC
LIMIT 1
```

## SKIPPED

- **USASpending**: No rate limit, unique constraints, bulk download preferred
- **CALC**: Multi-sort Elasticsearch workaround with dedup state, not amenable to simple page checkpointing
- **Awards/Exclusions/Subawards**: Not rate-limited enough to justify resume complexity
