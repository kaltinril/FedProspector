# Phase 44.4: USASpending Bulk Loader Fixes

## Context

The `load usaspending-bulk` CLI command fails with two distinct issues:

1. **403 Forbidden** on download for most fiscal years — the API returns a `file_url` immediately but the file isn't ready on USASpending's CDN yet. The code tries to download instantly, gets 403.
2. **Duplicate primary key** on FY2025 (which did download) — USASpending CSVs contain multiple rows per `generated_unique_award_id` (contract modifications), but `tmp_usaspending_bulk` inherits the PK constraint from `usaspending_award`, causing `LOAD DATA INFILE` to fail.

## Issue 1: 403 Forbidden on Bulk Download

### Root Cause

In [bulk_spending.py:81](fed_prospector/cli/bulk_spending.py#L81), when `request_bulk_download()` returns a `file_url` directly (line 466 of usaspending_client.py), the code skips polling and calls `download_bulk_file()` immediately. The file isn't actually available on USASpending's CDN yet — it returns 403.

FY2025 succeeded because it hit the `status_url` polling path (larger dataset = async generation), giving the CDN time to propagate the file.

### Fix

**In `usaspending_client.py`, `download_bulk_file()` (line 520):**

Add retry logic with backoff specifically for 403 errors. When a 403 is received, wait and retry (the file is being propagated to the CDN). Suggested approach:

```python
def download_bulk_file(self, file_url, dest_dir=None, max_retries=5, retry_delay=10):
    # ... existing setup code ...

    for attempt in range(max_retries + 1):
        response = self.session.get(file_url, stream=True, timeout=600)
        if response.status_code == 403 and attempt < max_retries:
            self.logger.warning(
                "Download returned 403 (attempt %d/%d), retrying in %ds...",
                attempt + 1, max_retries, retry_delay,
            )
            time.sleep(retry_delay)
            retry_delay = min(retry_delay * 2, 60)  # backoff, cap at 60s
            continue
        response.raise_for_status()
        break

    # ... existing download code ...
```

Also log the response body on non-retryable failures for debugging:
```python
if response.status_code != 200:
    self.logger.error("Download failed: %s %s", response.status_code, response.text[:500])
```

| File | Change |
|------|--------|
| [usaspending_client.py](fed_prospector/api_clients/usaspending_client.py) | Add 403 retry logic to `download_bulk_file()` (~line 520) |

## Issue 2: Duplicate Primary Key in Temp Table

### Root Cause

In [usaspending_bulk_loader.py:317](fed_prospector/etl/usaspending_bulk_loader.py#L317), `_create_temp_table()` creates the temp table with `CREATE TEMPORARY TABLE ... LIKE usaspending_award`, which copies the primary key constraint on `generated_unique_award_id`.

USASpending bulk CSVs contain **multiple rows per award ID** (one per contract modification/transaction). When `LOAD DATA INFILE` encounters a second row with the same `generated_unique_award_id`, it fails with a duplicate key error.

The upsert logic in `_upsert_from_temp()` already uses `ON DUPLICATE KEY UPDATE`, so it can handle deduplication — the problem is just getting the data into the temp table.

### Fix

**In `usaspending_bulk_loader.py`, `_create_temp_table()` (~line 317):**

Drop the primary key from the temp table after creation so `LOAD DATA INFILE` can load all rows including duplicates:

```python
def _create_temp_table(self, cursor):
    cursor.execute(
        "CREATE TEMPORARY TABLE IF NOT EXISTS tmp_usaspending_bulk "
        "LIKE usaspending_award"
    )
    cursor.execute("ALTER TABLE tmp_usaspending_bulk DROP PRIMARY KEY")
```

**In `_upsert_from_temp()` (~line 335):**

The existing `INSERT ... ON DUPLICATE KEY UPDATE` upsert already handles deduplication into the main table. However, if the temp table has multiple rows per award ID, the upsert will process them in insertion order — the last one wins. This is correct behavior (latest modification overwrites earlier ones), but verify the CSV rows are ordered chronologically. If not, add deduplication in the upsert query:

```sql
INSERT INTO usaspending_award (...)
SELECT ... FROM (
    SELECT *, ROW_NUMBER() OVER (
        PARTITION BY generated_unique_award_id
        ORDER BY last_modified_date DESC, modification_number DESC
    ) AS rn
    FROM tmp_usaspending_bulk
) ranked WHERE rn = 1
ON DUPLICATE KEY UPDATE ...
```

| File | Change |
