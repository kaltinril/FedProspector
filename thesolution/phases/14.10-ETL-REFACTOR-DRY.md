## Goal

Phase 14.9 just added identical staging logic to 7 loaders using copy-paste. This phase extracts the genuinely large shared patterns into reusable code. Focus is on real wins ‚Äî not every 3-line similarity.

**Out of scope deliberately:**
- Main batch loop template (too invasive, high breakage risk for moderate gain)
- Error handling block (saves ~140 lines but requires 7 parameters ‚Äî trades lines for complexity, not worth it)
- `_safe_get`, `str_or_none`, stats dict init (too small, not worth the abstraction)
- Hash field registry (metadata-as-code is fine where it is)
- Upsert templates (too loader-specific to generalize safely)

## The Three Wins

### Win 1 ‚Äî `StagingMixin`: Eliminate 350+ Lines of Copied Staging Code

**The problem:** Phase 14.9 added three nearly identical private methods to 6 loaders (entity is handled separately in Win 1b):

```
_extract_staging_key()   ‚Äî different per loader (API field names vary)
_insert_staging()        ‚Äî identical except table name + key column names
_mark_staging()          ‚Äî identical except table name
```

Plus the dual-connection setup block is repeated verbatim in every loader:
```python
stg_conn = get_connection()
stg_conn.autocommit = True
stg_cursor = stg_conn.cursor()
```

**The fix:** A `StagingMixin` class in `fed_prospector/etl/staging_mixin.py`.

```python
class StagingMixin:
    """
    Mixin for ETL loaders that write raw JSON to a staging table before normalizing.

    Subclass must define:
        _STG_TABLE: str              ‚Äî e.g. "stg_fpds_award_raw"
        _STG_KEY_COLS: list[str]     ‚Äî e.g. ["contract_id", "modification_number"]
        _extract_staging_key(raw: dict) -> dict   ‚Äî returns {col: val} for key cols
    """

    def _open_stg_conn(self):
        """Open a dedicated autocommit connection for staging writes."""
        conn = get_connection()
        conn.autocommit = True
        return conn, conn.cursor()

    def _insert_staging(self, stg_cursor, load_id: int, key_vals: dict, raw: dict) -> int:
        """Write raw JSON to staging table. Returns lastrowid."""
        raw_str = json.dumps(raw, sort_keys=True, default=str)
        raw_hash = hashlib.sha256(raw_str.encode()).hexdigest()
        col_names = ", ".join(["load_id"] + self._STG_KEY_COLS + ["raw_json", "raw_record_hash"])
        placeholders = ", ".join(["%s"] * (2 + len(self._STG_KEY_COLS) + 1))
        vals = [load_id] + [key_vals[c] for c in self._STG_KEY_COLS] + [raw_str, raw_hash]
        stg_cursor.execute(
            f"INSERT INTO {self._STG_TABLE} ({col_names}) VALUES ({placeholders})", vals
        )
        return stg_cursor.lastrowid

    def _mark_staging(self, stg_cursor, staging_id: int, processed: str, error_msg=None):
        """Update staging row outcome. processed='Y' (ok) or 'E' (error)."""
        stg_cursor.execute(
            f"UPDATE {self._STG_TABLE} SET processed=%s, error_message=%s WHERE id=%s",
            (processed, error_msg[:500] if error_msg else None, staging_id),
        )
```

**Per-loader changes** (6 loaders: awards, opportunity, exclusions, subaward, usaspending, fedhier):
- Add `StagingMixin` to class definition
- Remove `_insert_staging` and `_mark_staging` private methods (replaced by mixin)
- Add two class attributes: `_STG_TABLE` and `_STG_KEY_COLS`
- Keep `_extract_staging_key` ‚Äî it stays per-loader because the API field names vary

**Win 1b ‚Äî Entity Loader**: `stg_entity_raw` has a different schema (no auto-increment id, extra `processed_at` column, update keyed on `uei_sam + load_id`). Entity loader keeps its own `_insert_staging_entity` and `_mark_staging_entity` methods ‚Äî they're unique enough that forcing them through `StagingMixin` adds complexity, not removes it.

**Estimated savings:** ~350 lines across 6 loaders (remove ~60 lines per loader, add ~10 lines to each for class attributes + mixin import).

### Win 2 ‚Äî Shared `_parse_date` Utility: Eliminate 110 Lines of Diverged Date Parsing

**The problem:** 7 loaders each have their own `_parse_date` / `_norm_date` implementation. They handle overlapping but slightly different format sets. When a new date format appears in the API, someone picks one loader to fix and forgets the rest.

Current formats handled per loader:

| Loader | Formats handled |
|--------|----------------|
| awards | MM/DD/YYYY, YYYY-MM-DD |
| opportunity | YYYY-MM-DD, YYYYMMDD, MM/DD/YYYY, ISO 8601 T |
| exclusions | ISO, MM/DD/YYYY, MM-DD-YYYY |
| subaward | ISO, MM/DD/YYYY |
| usaspending | ISO, MM/DD/YYYY |
| entity | YYYY-MM-DD, YYYYMMDD, None guard |
| fedhier | ISO only |

**The fix:** A single `parse_date(value)` function in `fed_prospector/etl/etl_utils.py` (new file, or add to existing utils module) that handles the superset of all formats:

```python
from datetime import datetime

_DATE_FORMATS = [
    "%Y-%m-%d",           # ISO: 2024-01-15
    "%Y%m%d",             # compact: 20240115
    "%m/%d/%Y",           # US: 01/15/2024
    "%m-%d-%Y",           # dash-US: 01-15-2024
]

def parse_date(value) -> str | None:
    """
    Normalize any SAM.gov / USASpending date string to YYYY-MM-DD.
    Handles None, empty string, ISO 8601 with time component, and all common formats.
    Returns None on failure ‚Äî never raises.
    """
    if not value:
        return None
    s = str(value).strip()
    # Strip time component: "2024-01-15T00:00:00" ‚Üí "2024-01-15"
    if "T" in s:
        s = s.split("T")[0]
    for fmt in _DATE_FORMATS:
        try:
            return datetime.strptime(s, fmt).strftime("%Y-%m-%d")
        except ValueError:
            continue
    return None
```

**Per-loader changes:** Remove the local `_parse_date` / `_norm_date` method; replace calls with `parse_date(value)` imported from `etl_utils`.

**Estimated savings:** ~110 lines. Zero behavior change for valid dates; may silently fix bugs in loaders that were missing format coverage (e.g. awards loader didn't handle `YYYYMMDD`).

### Win 3 ‚Äî Shared `_parse_decimal` Utility: Eliminate 60 Lines of Copied Amount Parsing

**The problem:** 5 loaders (awards, opportunity, usaspending, subaward, entity) each implement a `_parse_decimal` / `_parse_amount` method with the same logic: handle None, strip commas, cast to float or Decimal.

**The fix:** Add `parse_decimal(value)` to `fed_prospector/etl/etl_utils.py`:

```python
from decimal import Decimal, InvalidOperation

def parse_decimal(value) -> str | None:
    """
    Normalize a dollar amount to a decimal string MySQL can store as DECIMAL.
    Strips commas, handles None and non-numeric. Returns None on failure.
    """
    if value is None:
        return None
    try:
        return str(Decimal(str(value).replace(",", "").strip()))
    except InvalidOperation:
        return None
```

**Per-loader changes:** Remove local `_parse_decimal` / `_parse_amount`; replace with `parse_decimal(value)` from `etl_utils`.

**Estimated savings:** ~60 lines.

### Win 4 ‚Äî `fetch_existing_hashes`: Eliminate 150 Lines of Copied Hash-Fetch Boilerplate

**The problem:** 6 loaders each implement `_get_existing_hashes()` using the exact same structure: open a connection, run `SELECT key_col, record_hash FROM table WHERE record_hash IS NOT NULL`, close it, return a dict. Only the table name and key column differ. This is pure boilerplate.

Affected loaders: awards, opportunity, exclusions, subaward, usaspending, fedhier (~25 lines each).

**The fix:** Add `fetch_existing_hashes(table_name, key_col)` to `fed_prospector/etl/etl_utils.py`:

```python
def fetch_existing_hashes(table_name: str, key_col: str) -> dict:
    """
    Returns a {key_value: record_hash} dict for all rows in table_name
    that already have a record_hash. Used for change-detection in ETL loaders.
    """
    conn = get_connection()
    cursor = conn.cursor()
    try:
        cursor.execute(
            f"SELECT {key_col}, record_hash FROM {table_name} WHERE record_hash IS NOT NULL"
        )
        return {row[0]: row[1] for row in cursor.fetchall()}
    finally:
        cursor.close()
        conn.close()
```

**Per-loader changes:** Remove the `_get_existing_hashes` method; replace the one call site with `fetch_existing_hashes(self._TABLE, self._KEY_COL)` (or pass table/col directly ‚Äî whatever is cleaner in context).

**Note on dynamic SQL:** `table_name` and `key_col` come from class-level string constants in trusted loader code, not user input. No injection risk.

**Estimated savings:** ~150 lines across 6 loaders.

### Win 5 ‚Äî Cursor Context Manager: Eliminate 50+ Lines of try/finally in DB Layer

**The problem:** `db/connection.py` provides `get_connection()` but no context manager. Every consumer manually repeats:
```python
conn = get_connection()
cursor = conn.cursor(dictionary=True)
try:
    ...
finally:
    cursor.close()
    conn.close()
```

This pattern appears ~27 times across `health_check.py`, `db_maintenance.py`, `scheduler.py`, `prospect_manager.py`, and the loaders themselves. It's foundational boilerplate that makes every reader/query method longer than it needs to be.

**The fix:** Add two context managers to `fed_prospector/db/connection.py`:
```python
from contextlib import contextmanager

@contextmanager
def get_cursor(dictionary=False):
    """Context manager that yields a cursor and closes conn on exit."""
    conn = get_pool().get_connection()
    cursor = conn.cursor(dictionary=dictionary)
    try:
        yield cursor
    finally:
        cursor.close()
        conn.close()
```

Usage replaces 8-line blocks with 2 lines:
```python
with get_cursor(dictionary=True) as cursor:
    cursor.execute("SELECT ...")
    return cursor.fetchall()
```

**Scope for this phase:** Update `db/connection.py` only ‚Äî add the context manager. The CLI and other files will adopt it in Phase 14.11. Loaders that use batch-commit connections cannot use this pattern (they need to control commit timing) ‚Äî those stay as-is.

**Estimated savings (phase 14.10 scope):** ~20 lines in `db_maintenance.py` and `health_check.py` read-only query methods. Full payoff comes in Phase 14.11 when CLI adopts it.
