# Phase 44.3: Data Enrichment Opportunities

**Status**: PLANNED
**Depends on**: Phase 44.2 (Resource Link Metadata)

## Context

Audit of all data tables and SAM.gov API responses to identify fields we're not capturing, codes we're not resolving, and cross-entity links we're not making. Goal: maximize the intelligence value of data we already have or can get cheaply.

## Audit Findings

### High Value -- Missing Data We Already Receive

These fields are in the SAM.gov API response but we're not storing them.

#### 1. Point of Contact (POC) Data -- NOT BEING LOADED

**Tables exist but are empty**: `contracting_officer` (0 rows), `opportunity_poc` (0 rows).

The SAM.gov API returns full POC data on every opportunity:
```json
"pointOfContact": [
  {"type": "primary", "fullName": "Anthony Plath", "email": "anthony.plath@usmc.mil", "phone": "", "fax": ""},
  {"type": "secondary", "fullName": "Eric G. Daniels", "email": "Eric.G.Daniels@usmc.mil", "phone": ""}
]
```

**Impact**: Users can't see who to contact about an opportunity. This is critical for business development.
**Fix**: The loader's `_normalize_opportunity()` ignores `pointOfContact`. Add extraction to insert into `contracting_officer` + `opportunity_poc`.
**Effort**: ~0.5 days. Schema exists, just need loader code.

#### 2. Office Address -- NOT BEING STORED

SAM.gov returns the contracting office address:
```json
"officeAddress": {"city": "QUANTICO", "state": "VA", "zipcode": "22134-6050", "countryCode": "USA"}
```

Our `opportunity` table has an `office_address` column name in the schema but it's not being populated by the loader. The column doesn't actually exist -- we'd need to either add it or store this in the existing `office` field.

**Impact**: Useful for geographic analysis and filtering.
**Fix**: Add columns or store as JSON. Update loader.
**Effort**: ~0.25 days.

#### 3. Awardee Location Data -- NOT BEING STORED

For awarded opportunities, SAM.gov returns the awardee's location:
```json
"award": {"awardee": {"location": {"city": {"name": "Huntsville"}, "state": {"code": "AL"}, "country": {"code": "USA"}, "zip": "35805"}, "cageCode": "7ERY6"}}
```

We store `awardee_uei` and `awardee_name` but not the location or CAGE code.

**Impact**: Competitor location analysis without needing entity table lookups.
**Fix**: Add columns: `awardee_cage_code`, `awardee_city`, `awardee_state`, `awardee_zip`.
**Effort**: ~0.25 days.

#### 4. Full Parent Path -- PARSED BUT LOSSY

We parse `fullParentPathName` ("DEPT OF DEFENSE.DEPT OF THE NAVY.USMC.MARCOR SYSCOM.COMMANDER") into `department_name`, `sub_tier`, and `office`. But the middle segments are lost. The full hierarchy is valuable for filtering by command/agency.

**Impact**: Can't filter by sub-commands or intermediate hierarchy levels.
**Fix**: Store `full_parent_path_name` and `full_parent_path_code` as raw strings. The parsed fields remain for display.
**Effort**: ~0.25 days.

### Medium Value -- Cross-Entity Enrichment

#### 5. Incumbent/Awardee Entity Linking

`opportunity.incumbent_uei` and `opportunity.awardee_uei` exist but are always NULL (0 rows populated). Meanwhile, `entity` has 793K records with UEI. Even when populated, there's no FK or JOIN being done.

**Impact**: Can't show incumbent details, SBA certifications, or business type on opportunity views.
**Fix**: Two parts: (a) ensure loader populates UEI fields from API data, (b) C# API JOINs to entity table when displaying opportunity detail.
**Effort**: ~0.5 days.

#### 6. Contracting Office to Federal Hierarchy Linking

`opportunity.contracting_office_id` is populated (3,256 of 14,241 rows) but never joined to `federal_organization` (887 rows). The codes don't appear to match cleanly (opportunity stores the last code segment, federal_organization uses `agency_code`).

**Impact**: Can't show full agency details, org descriptions, or hierarchy navigation from opportunities.
**Fix**: Investigate code matching. May need to store `fullParentPathCode` segments and join on `agency_code` or `cgac`.
**Effort**: ~0.5 days (investigation) + 0.25 days (implementation).

#### 7. Period of Performance -- COLUMN EXISTS, NEVER POPULATED

`period_of_performance_start` and `period_of_performance_end` columns exist but are always NULL. Need to check if SAM.gov API returns this data -- it may only be available on some opportunity types.

**Fix**: Check raw staging data for these fields. If present, add to loader.
**Effort**: ~0.25 days.

#### 8. Estimated Contract Value -- COLUMN EXISTS, NEVER POPULATED

`estimated_contract_value` is always NULL. SAM.gov may not provide this on solicitations (only on awards via `award.amount`).

**Fix**: Check if any raw API responses contain estimated values. May be a field that simply isn't returned by the API for pre-award opportunities.
**Effort**: ~0.25 days investigation.

### Low Value -- Nice to Have

#### 9. Description Text Caching

`description_url` stores a URL to the description endpoint. The actual text is fetched on-demand by the C# API. Could cache the text in a column to avoid repeated API calls and enable full-text search.

**Impact**: Enables opportunity full-text search. Reduces API dependency.
**Fix**: Add `description_text` TEXT column. Fetch during ETL load (no rate limit on this endpoint).
**Effort**: ~0.5 days.

#### 10. Multiple NAICS Codes

SAM.gov returns `naicsCodes` (array) in addition to `naicsCode` (single). We only store the single code. Some opportunities have multiple NAICS codes.

**Fix**: Store as JSON array or child table.
**Effort**: ~0.25 days.

### Already Working Well

These areas are fine -- no enrichment needed:

- **NAICS code descriptions**: Reference table joins work (verified: 336330 -> "Motor Vehicle Steering and Suspension Components...")
- **PSC code descriptions**: Reference table joins work (verified: 2530 -> "VEHICULAR BRAKE, STEERING, AXLE, WHEEL, AND TRACK COMPONENTS")
- **Set-aside descriptions**: Stored directly from API (e.g., "Total Small Business Set-Aside (FAR 19.5)")
- **Entity data**: Comprehensive (793K entities with addresses, business types, NAICS, SBA certs, POCs)
- **Resource link metadata**: Being added by Phase 44.2

## Recommended Priority

### Batch 1 -- Quick wins, high value (1-2 days total)
1. **POC data loading** -- Tables exist, just need loader code. Critical for BD.
2. **Awardee location + CAGE code** -- 4 columns, straightforward.
3. **Full parent path storage** -- 2 columns, no logic change.
4. **Period of performance** -- Investigate + populate if API provides it.

5. **Incumbent/awardee entity linking** -- Populate UEI fields + API JOINs.
6. **Federal hierarchy linking** -- Code matching investigation.
7. **Office address** -- Schema change + loader update.
8. **Estimated contract value investigation** -- May not be available pre-award.

### Batch 3 -- Larger effort, nice to have
9. **Description text caching** -- Enables full-text search.
10. **Multiple NAICS codes** -- Low impact unless users need multi-NAICS filtering.
