## CRITICAL Fixes

### Fix 1 — Exclusions Change Detection Completely Broken

**File:** `fed_prospector/etl/exclusions_loader.py:117`

**Problem:** `fetch_existing_hashes("sam_exclusion", "uei")` returns a dict keyed on `uei` values. But the lookup at line 145 uses `existing_hashes.get(record_key)` where `record_key` is the composite string `"{uei}|{activation_date}|{exclusion_type}"` built by `_make_exclusion_key`. A bare UEI like `"ABCDE12345"` never matches a composite key like `"ABCDE12345|2024-01-15|Ineligible"`. Result: `old_hash` is always `None`, every record appears new on every run, `records_unchanged` is always 0, and every load re-inserts data that hasn't changed.

**Fix:** `fetch_existing_hashes` only supports single-column keys. For exclusions the natural composite key (`uei|activation_date|exclusion_type`) cannot be expressed as a single column lookup. Replace the call with an inline query that fetches the composite key correctly:

```python
# Remove: existing_hashes = fetch_existing_hashes("sam_exclusion", "uei")
# Replace with:
conn_h = get_connection()
cur_h = conn_h.cursor()
try:
    cur_h.execute(
        "SELECT CONCAT(COALESCE(uei,''), '|', COALESCE(activation_date,''), '|', COALESCE(exclusion_type,'')), record_hash "
        "FROM sam_exclusion WHERE record_hash IS NOT NULL"
    )
    existing_hashes = {row[0]: row[1] for row in cur_h.fetchall()}
finally:
    cur_h.close()
    conn_h.close()
```

**Verification:** Run `load exclusions` twice with the same data. Second run should show `records_unchanged > 0` and `records_inserted = 0` (for unchanged records).

### Fix 2 — Subaward Change Detection Broken for Primes with Multiple Subawards

**File:** `fed_prospector/etl/subaward_loader.py:117`

**Problem:** `fetch_existing_hashes("sam_subaward", "prime_piid")` keys the hash dict on `prime_piid` alone. A prime with multiple subawards produces multiple rows sharing the same `prime_piid`. The dict only keeps one hash per prime — the last one fetched. All other subawards for that prime get `old_hash` = some other subaward's hash, causing incorrect "unchanged" classification and silently skipping real updates. The `_make_subaward_key` helper exists but is never used in the load loop.

**Fix:** Replace the single-column lookup with a composite key query:

```python
# Remove: existing_hashes = fetch_existing_hashes("sam_subaward", "prime_piid")
# Replace with:
conn_h = get_connection()
cur_h = conn_h.cursor()
try:
    cur_h.execute(
        "SELECT CONCAT(COALESCE(prime_piid,''), '|', COALESCE(sub_uei,'')), record_hash "
        "FROM sam_subaward WHERE record_hash IS NOT NULL"
    )
    existing_hashes = {row[0]: row[1] for row in cur_h.fetchall()}
finally:
    cur_h.close()
    conn_h.close()
```

Then update the `record_key` lookup in the main loop to use the same composite key:
```python
record_key = f"{normalized.get('prime_piid', '')}|{normalized.get('sub_uei', '')}"
old_hash = existing_hashes.get(record_key)
```

### Fix 3 — `entity_loader.py` Staging Connection Leak on Init Failure

**File:** `fed_prospector/etl/entity_loader.py:182-187`

**Problem:** The staging connection is opened with three sequential statements:
```python
stg_conn = get_connection()
stg_conn.autocommit = True
stg_cursor = stg_conn.cursor()
```
If `stg_conn.cursor()` raises (e.g., the connection drops mid-setup), `stg_conn` is open but `stg_cursor` is unbound. The `finally` block at lines 308-313 attempts `stg_cursor.close()` and raises `UnboundLocalError`, masking the original exception and leaving `stg_conn` open (leaks the pool slot).

**Fix:** Mirror the `StagingMixin._open_stg_conn()` pattern — open both atomically before the try:

```python
if write_staging:
    stg_conn, stg_cursor = None, None
    try:
        stg_conn = get_connection()
        stg_conn.autocommit = True
        stg_cursor = stg_conn.cursor()
    except Exception:
        if stg_conn:
            stg_conn.close()
        raise
else:
    stg_conn, stg_cursor = None, None
```

And update the `finally` block to guard with `if stg_conn:` and `if stg_cursor:` before closing.

## HIGH Fixes

### Fix 4 — `len()` on Generator Input Crashes All Loaders

**Files:** `awards_loader.py:107`, `opportunity_loader.py:76`, `usaspending_loader.py:86`, `exclusions_loader.py:103`, `subaward_loader.py:103`, `fedhier_loader.py:92`

**Problem:** All 6 loaders call `len(data)` in the opening log line:
```python
self.logger.info("Starting load (%d records, load_id=%d)", len(data), load_id)
```
If `data` is a generator (which API client `search_X_all()` methods return), `len()` raises `TypeError: object of type 'generator' has no len()` before any records are processed. The load fails immediately with a confusing error.

**Fix:** Replace `len(data)` with a safe alternative in all 6 loaders:
```python
# Before any iteration:
data = list(data)  # materialize generator to allow len() and re-iteration
self.logger.info("Starting load (%d records, load_id=%d)", len(data), load_id)
```
Or if memory is a concern for large datasets, log without a count:
```python
self.logger.info("Starting load (load_id=%d)", load_id)
```
The list-materialization approach is simpler and consistent with how existing loaders work (they already iterate data twice in some paths).

### Fix 5 — `fedhier_loader.py` full_refresh Not Atomic

**File:** `fed_prospector/etl/fedhier_loader.py:216-239`

**Problem:** `full_refresh()` commits a `TRUNCATE TABLE federal_organization` before calling `load_organizations()`. If `load_organizations()` then fails (DB connection loss, bad record, etc.), the table stays empty. No rollback is possible after the TRUNCATE commit.

**Fix:** Do not commit the TRUNCATE separately. Hold the TRUNCATE in the same transaction as the load:
```python
def full_refresh(self, orgs_data, load_id):
    conn = get_connection()
    cursor = conn.cursor()
    try:
        cursor.execute("TRUNCATE TABLE federal_organization")
        # Do NOT commit here — let load_organizations() commit after data is in
        cursor.close()
        conn.close()
        return self.load_organizations(orgs_data, load_id)
    except Exception:
        conn.rollback()
        cursor.close()
        conn.close()
        raise
```
Note: InnoDB TRUNCATE TABLE acquires a metadata lock and implicitly commits in MySQL — this is a MySQL limitation. Document this as a known risk in the code comment. As a practical mitigation, consider using `DELETE FROM federal_organization` instead of TRUNCATE (slower but transactional), or accept the risk given this is a rarely-run full-refresh operation on internal hierarchy data.

### Fix 6 — `bulk_loader.py` FK Checks Not Restored on Exception

**File:** `fed_prospector/etl/bulk_loader.py:385-419`

**Problem:** For `load_type == "FULL"`, FK checks are disabled with `SET FOREIGN_KEY_CHECKS = 0` then re-enabled with `SET FOREIGN_KEY_CHECKS = 1` — but only on the success path. On exception, `conn.rollback()` is called without first re-enabling FK checks. Since `FOREIGN_KEY_CHECKS` is a session variable, it resets when the connection closes — but if the connection pool recycles without closing (depending on pool config), the next user of that connection has FK checks silently disabled.

**Fix:** Move `SET FOREIGN_KEY_CHECKS = 1` into the `finally` block:
```python
finally:
    if load_type == "FULL":
        try:
            cursor.execute("SET FOREIGN_KEY_CHECKS = 1")
            conn.commit()
        except Exception:
            pass  # Best-effort; connection may be broken
    cursor.close()
    conn.close()
```

### Fix 7 — Timezone Dropped from `response_deadline` in Opportunity Loader

**File:** `fed_prospector/etl/opportunity_loader.py:440-452`

**Problem:** `_parse_datetime()` strips the timezone from SAM.gov datetime strings (e.g. `"2026-06-30T23:59:00-04:00"`) by calling `strftime("%Y-%m-%d %H:%M:%S")` which discards the offset. SAM.gov timestamps are Eastern time. A deadline of `23:59 EDT` stored as `23:59` local will appear as 5 hours later than it actually is for UTC servers. The `open_only` filter in `run_search` compares this stored time against `NOW()`, causing opportunities to appear open several hours past their actual close.

**Fix:** Normalize to UTC before storing:
```python
from datetime import timezone
# After parsing with %z:
dt = datetime.strptime(s, fmt)  # aware datetime
dt_utc = dt.astimezone(timezone.utc)
return dt_utc.strftime("%Y-%m-%d %H:%M:%S")  # store as UTC
```
Add a comment noting all `response_deadline` values are stored in UTC.

### Fix 8 — `usaspending_loader.py` `last_modified_date` Mapped to Wrong Field

**Problem:**
```python
"last_modified_date": parse_date(raw.get("Last Date to Order")),
```

`"Last Date to Order"` is the period-of-performance end date for IDV contracts, not a modification date. Storing a PoP end date as `last_modified_date` pollutes change detection and audit information.

**Fix:** Map to the correct field or leave as None until enrichment fills it:
```python
"last_modified_date": parse_date(raw.get("Last Modified Date")),
```

### Fix 9 — `dat_parser.py` Malformed Dates Silently Become NULL

**File:** `fed_prospector/etl/dat_parser.py:425-439`

**Problem:** `_norm_date()` returns `None` with no log warning for any date that is not exactly 8 digits. A date like `"2026/02/01"` or `"20260201 "` (trailing space) silently becomes NULL. Important fields (`registration_expiration_date`, `initial_registration_date`) are lost with no trace.

**Fix:** Add a warning log on the fallback path:
```python
def _norm_date(value):
    if not value or not value.strip():
        return None
    s = value.strip()
    if len(s) == 8 and s.isdigit():
        return f"{s[:4]}-{s[4:6]}-{s[6:8]}"
    logger.warning("dat_parser: unrecognized date format %r — stored as NULL", s)
    return None
```

## MEDIUM Fixes

### Fix 10 — Exclusions Staging Key Reads Wrong Fields

**File:** `fed_prospector/etl/exclusions_loader.py:431-434`

**Problem:** `_extract_staging_key()` reads `uei`, `activationDate`, `exclusionType` from top-level keys — but the SAM.gov exclusions API nests them under `exclusionIdentification.ueiSAM`, `exclusionActions.listOfActions[0].activateDate`, etc. All staging rows get `record_id = "||"` (empty composite), making staging useless for replay/debugging.

**Fix:** Update `_extract_staging_key()` to mirror the same field extraction logic used in `_normalize_exclusion()`:
```python
def _extract_staging_key(self, raw):
    eid = raw.get("exclusionIdentification", {})
    actions = raw.get("exclusionActions", {}).get("listOfActions", [{}])
    uei = eid.get("ueiSAM") or eid.get("cageCode") or eid.get("npi") or ""
    act_date = actions[0].get("activateDate", "") if actions else ""
    ex_type = raw.get("exclusionType", {}).get("value", "") if isinstance(raw.get("exclusionType"), dict) else raw.get("exclusionType", "")
    composite = f"{uei}|{act_date}|{ex_type}"
    return {"record_id": composite[:100]}
```

### Fix 11 — `health_check.py` Double Snapshot on External Call

**File:** `fed_prospector/etl/health_check.py:54-63`

**Problem:** `save_snapshot(results=None)` calls `check_all()` which calls `save_snapshot(results)`. When `save_snapshot()` is called externally without arguments, two DB rows are written.

**Fix:** Remove the `save_snapshot()` call from inside `check_all()`. Let callers decide whether to save:
```python
def check_all(self):
    results = { ... }
    return results  # Caller decides whether to call save_snapshot(results)
```
Update all call sites to:
```python
results = checker.check_all()
checker.save_snapshot(results)
```
