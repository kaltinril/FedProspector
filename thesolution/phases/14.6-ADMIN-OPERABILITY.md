# Phase 14.6: Admin Operability & CLI Hardening

**Status**: COMPLETE -- 2026-03-02
**Dependencies**: Phase 14.5 (Multi-Tenancy), Phase 6 (Automation)
**Deliverable**: First-time setup wizard, automated entity loading, schedule installer, log rotation, admin CLI commands, job history, service manager hardening, API key tracking
**Repository**: `fed_prospector/`, `fed_prospector.py`

---

> **BLOCKER FOR UI PHASES**: This phase ensures an admin can stand up, operate, and troubleshoot the system without AI assistance or raw SQL. Discovered during pre-UI audit on 2026-03-02 — the system works once running but first-time setup is rough, daily operations require tribal knowledge, and several admin tasks have no CLI fallback.

## Overview

Audit of the admin experience revealed 16 gaps across 5 categories. This phase addresses them in 3 sprints ordered by severity. The ETL-to-database-to-App-API data pipeline is fully aligned (19 tables, all column mappings correct, multi-tenancy properly implemented), so no data layer changes are needed.

**Key principle**: Everything must be runnable without AI. CLI commands with clear names, `--help` support, and actionable error messages.

## Sprint 1: Critical — Unblock Daily Admin Use

### S1.1: Entity Auto-Load Wrapper

**Problem**: Entity loading is a broken 2-step process (`download-extract` then manual file copy to `load-entities --file`). Cannot be automated by `catchup-datasets`.

**Solution**: New CLI command `refresh-entities` that wraps both steps:

```bash
python main.py refresh-entities --type daily     # Download + load daily extract
python main.py refresh-entities --type monthly    # Download + load monthly extract
```

**Implementation**:
- New command in `cli/entities.py`
- Calls `SAMExtractClient.download_daily_extract()`, captures file path
- Passes file path to `BulkLoader.bulk_load_entities()` or JSON loader
- Logs to `etl_load_log` with `source_system='SAM_ENTITY'`
- Update scheduler `entity_daily` job command to use `refresh-entities --type daily`
- Set `catchup_safe: True` for entity_daily job

**Files**: `cli/entities.py`, `etl/scheduler.py`, `main.py`

### S1.2: Verify Setup Command

**Problem**: No way to check if all prerequisites are met. Admin gets cryptic Python tracebacks when bcrypt is missing, MySQL path is wrong, or .env is incomplete.

**Solution**: New CLI command `verify-setup`:

```bash
python main.py verify-setup
```

**Checks** (in order):
1. Python version >= 3.14
2. Required pip packages installed (bcrypt, mysql-connector-python, click, python-dotenv, requests)
3. `.env` file exists with all required keys (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD, SAM_API_KEY)
4. MySQL binary found at `MYSQL_BIN_DIR` or on PATH
5. MySQL server reachable (attempt connection)
6. Database `fed_contracts` exists
7. Schema tables present (count vs expected 56)
8. SAM.gov API key valid (optional, uses 1 call)

**Output**: Checkmark/X for each item with actionable fix instructions on failure.

**Files**: New `cli/setup.py`, `main.py`

### S1.3: Log Rotation

**Problem**: Single log file grows forever. Using basic `FileHandler`.

**Solution**: Switch to `RotatingFileHandler` in `config/logging_config.py`:
- Max 50 MB per file
- Keep 10 backup files (500 MB total max)
- UTF-8 encoding

**Files**: `config/logging_config.py`

### S1.4: Setup Schedule Command

**Problem**: Admin must copy-paste 8 `schtasks` commands manually. No Linux/cron support.

**Solution**: New CLI command `setup-schedule`:

```bash
python main.py setup-schedule                    # Auto-detect OS, create tasks
python main.py setup-schedule --platform windows  # Force Windows Task Scheduler
python main.py setup-schedule --platform linux    # Generate crontab entries
python main.py setup-schedule --dry-run           # Show what would be created
python main.py setup-schedule --remove            # Remove all scheduled tasks
```

**Implementation**:
- Reads `JOBS` dict from `etl/scheduler.py` (already has schedule metadata)
- Windows: generates and runs `schtasks /create` commands
- Linux: generates crontab entries and offers to install via `crontab -e`
- `--dry-run`: prints commands without executing
- `--remove`: deletes created tasks/cron entries

**Files**: New `cli/schedule_setup.py`, `main.py`

## Sprint 2: High — Unblock User Management & History

### S2.1: Admin CLI Commands

**Problem**: After bootstrap, creating orgs, inviting users, disabling users, resetting passwords all require API/curl. No CLI fallback before UI exists.

**New commands** (all in `cli/admin.py`):

```bash
# Organization management
python main.py create-org --name "Acme Corp" --slug acme-corp
python main.py list-orgs

# User invitation (returns invite code for manual sharing)
python main.py invite-user --email user@acme.com --org-id 2 --role member

# User management
python main.py list-org-members --org-id 2
python main.py disable-user --user-id 5
python main.py enable-user --user-id 5
python main.py reset-password --user-id 5     # Prints temp password
```

**Implementation notes**:
- All use direct MySQL via `db/connection.py` (same as `create-sysadmin`)
- Password hashing via `utils/password.py` (BCrypt Enhanced, compatible with C# API)
- `reset-password` sets `force_password_change='Y'`, revokes sessions
- `invite-user` creates row in `organization_invite` with SHA-256 invite code + 7-day expiry

**Files**: `cli/admin.py`, `main.py`

### S2.2: Job/Load History CLI

**Problem**: Can't see past load runs without raw SQL queries.

**New commands** (in `cli/health.py`):

```bash
python main.py load-history                          # Last 20 loads, all sources
python main.py load-history --source SAM_OPPORTUNITY  # Filter by source
python main.py load-history --days 7                  # Last 7 days
python main.py load-history --status FAILED           # Only failures
python main.py load-history --limit 50                # More results
```

**Output**: Table with columns: started_at, source, status, duration, records_in, records_out, errors

**Files**: `cli/health.py`, `main.py`

### S2.3: Service Manager Hardening

**Problem**: Port conflicts, MySQL binary not found, and timeouts produce useless error messages.

**Improvements to `fed_prospector.py`**:

1. **Pre-flight checks before start**:
   - Verify MySQL binary exists at configured path
   - Check port 3306 availability before starting MySQL
   - Check port 5056 availability before starting API
   - Print process info if port already in use

2. **Better timeout diagnostics**:
   - At 10s: "Still connecting..."
   - At 20s: "Still waiting, checking common issues..."
   - On timeout: List possible causes (binary not found, port in use, corrupted data dir)

3. **Build error handling**:
   - Don't `sys.exit(1)` on build failure when building `all`
   - Continue to next service, report failures at end

**Files**: `fed_prospector.py`

## Sprint 3: Production Readiness

### S3.1: API Key Expiration Tracking

**Problem**: SAM.gov keys expire every 90 days silently. No warning.

**Solution**:
- Add `SAM_API_KEY_CREATED` and `SAM_API_KEY_2_CREATED` to `.env` (ISO date)
- In `check-health`, compute days remaining. Warn at < 14 days, error at < 3 days.
- Add `check-api-keys` output to show expiration status

**Files**: `etl/health_check.py`, `.env`, `config/settings.py`

### S3.2: Health Check Persistence

**Problem**: Scheduled health check runs daily but output only goes to console/log. No alerting.

**Solution**:
- New table `etl_health_snapshot` to store check results as JSON
- `check-health` writes results to this table
- `GET /api/v1/admin/etl-status` already exists in C# API — wire it to read from `etl_health_snapshot`
- Future: email/Slack webhook integration (deferred to Phase 70)

**Files**: `db/schema/tables/50_etl.sql`, `etl/health_check.py`, `cli/health.py`

### S3.3: Seed Quality Rules in Build

**Problem**: `seed-quality-rules` is a silent dependency. Admin doesn't know to run it.

**Solution**:
- `build-database` auto-runs `seed-quality-rules` after table creation
- `verify-setup` checks if rules are seeded

**Files**: `cli/database.py`

## Previously Fixed (This Session)

These bugs were discovered and fixed during the audit that led to this phase:

1. **SAM_OPPORTUNITY_KEY2 mismatch** — Health checks looked for `SAM_OPPORTUNITY_KEY2` but ETL logged `SAM_OPPORTUNITY`. Fixed across HealthController, AdminService, health_check.py, scheduler.py, and tests.

2. **Release/Debug build mismatch** — Service manager built Release but ran Debug. Added `-c Release` to start command.

3. **UTC vs local time** — C# HealthController used `DateTime.UtcNow` but Python ETL stores local time. Changed to `DateTime.Now` in HealthController and AdminService.

4. **Solution file name** — `fed_prospector.py` referenced `FedProspector.Api.slnx` instead of `FedProspector.slnx`.

5. **catchup-datasets** — New command to auto-refresh stale data sources.

6. **Budget exhaustion warnings** — All paginated loaders now warn when `--max-calls` budget runs out before all data is fetched.

7. **Streaming output** — `catchup-datasets` streams job output in real-time instead of buffering.

8. **estimated_api_calls** — Job definitions now include API call estimates for `--dry-run` display.
