# Phase 44.6: Schema Drift Remediation

## Status: COMPLETE
## Priority: HIGH — blocks clean DB rebuilds

## Problem Statement

An audit revealed that the live database has drifted from the DDL scripts in `fed_prospector/db/schema/`. The DDL scripts are correct (they represent the intended schema); the live DB is behind. A fresh `build-database` produces a different schema than what's running.

## Root Cause

**Phase 14.14 (Database Schema Fixes)** correctly updated all DDL files with index additions, type corrections, and constraint changes. It also created a migration file (`fed_prospector/db/schema/migrations/phase14_14_schema_fixes.sql`) with the corresponding ALTER TABLE statements.

**Phase 14.24 (DDL Consolidation)** then deleted all migration files in favor of "single source of truth" DDL scripts — but assumed the migrations had already been applied to the live DB. They had not. The ALTER TABLE statements were lost.

## Methodology

This audit used three approaches:
1. **Phase document review** — Read all 44.x phase docs for schema changes
2. **Test DB rebuild** — Created `fed_contracts_test` from DDL scripts, compared against `fed_contracts`
3. **Built-in schema checker** — Ran `python main.py health check-schema --fix`

All three approaches confirmed the same findings.

## Finding 1: Missing Tables (DDL defines them, live DB does not have them)

### 1a. `data_load_request` — Phase 43 (On-Demand Award Loading)

| Attribute | Detail |
|-----------|--------|
| DDL File | `tables/55_data_load_request.sql` |
| Introduced | Phase 43 (On-Demand Award Loading) — COMPLETE |
| Purpose | Cross-process message queue between C# API and Python ETL |
| Status | **Fully wired up, actively used code on both sides** |

**How it works**: When a user views an award not in the local DB, the C# API inserts a `PENDING` row. The Python `DemandLoader` polls the table, fetches data from USASpending.gov/SAM.gov, loads it, and marks the row `COMPLETED` or `FAILED`. Supports two request types: `USASPENDING_AWARD` (quick, no rate limits) and `FPDS_AWARD` (enrichment, rate-limited).

**Code references**:
| File | Role |
|------|------|
| `fed_prospector/etl/demand_loader.py` | `DemandLoader` — polls PENDING, processes, updates status |
| `fed_prospector/cli/demand.py` | `process-requests` CLI (`--watch` for continuous polling) |
| `api/.../Services/AwardService.cs` | `RequestLoadAsync()` inserts PENDING rows; `GetLoadStatusAsync()` reads status |
| `api/.../Controllers/AwardsController.cs` | `POST {contractId}/load` and `GET {contractId}/load-status` endpoints |
| `api/.../Models/DataLoadRequest.cs` | EF Core entity model |

**Impact**: The on-demand loading feature silently fails — API inserts succeed only if the table exists.

**Verified active**: Phase 43 is NOT superseded. Phase 44 explicitly depends on it (line 4: "Depends on: Phase 43"). The auto-FPDS enrichment that would have replaced on-demand loading (Phase 44B) was deferred to Phase 500A (backlog). The full chain is wired end-to-end:
- UI: "Load This Award" button in `AwardDetailPage.tsx` (lines 574-617) when `dataStatus === 'not_loaded'`, polls every 4s
- API: `POST {contractId}/load` and `GET {contractId}/load-status` in `AwardsController.cs` (lines 57-70)
- Service: `AwardService.RequestLoadAsync()` inserts PENDING rows (line 534), `GetLoadStatusAsync()` reads status (line 576)
- Python: `DemandLoader` in `demand_loader.py` polls PENDING, fetches from USASpending/SAM.gov, marks COMPLETED/FAILED
- CLI: `python main.py demand process-requests --watch` (continuous polling every 5s)

Without this table, clicking "Load This Award" in the UI will error out.

## Finding 2: Missing Indexes — Phase 14.14 (Database Schema Fixes)

All four missing indexes were introduced in **Phase 14.14**, all marked HIGH priority.

| Table | Index Name | Columns | Type | Why It Matters |
|-------|-----------|---------|------|----------------|
| `fpds_contract` | `idx_fpds_ultimate_completion` | `ultimate_completion_date` | Regular | Used by `v_procurement_intelligence` view for re-compete prediction. Without it, queries filtering on contract expiration do full table scans. |
| `fpds_contract` | `idx_fpds_idv_piid` | `idv_piid` | Regular (prefix 50) | Links task orders to parent IDV contracts. "Who won work under this vehicle?" queries scan entire table without it. Used by `awards_loader.py` and `calc_loader.py`. |
| `sam_exclusion` | `uk_excl_key` | `uei, exclusion_type, activation_date` | **Unique** | Both performance AND data integrity. `exclusions_loader._upsert_exclusion()` does a SELECT-then-INSERT pattern on these 3 columns — without the unique index, duplicates can accumulate on crash. Note: MySQL treats NULLs as distinct in UNIQUE keys, so NULL-UEI exclusions won't be deduplicated (documented known limitation). |
| `usaspending_award` | `idx_usa_modified` | `last_modified_date` | Regular | Incremental loads filter by `last_modified_date`. Table grows to hundreds of thousands of rows; full scans are expensive. |

### Fix SQL
```sql
ALTER TABLE `fpds_contract` ADD INDEX `idx_fpds_ultimate_completion` (`ultimate_completion_date`);
ALTER TABLE `fpds_contract` ADD INDEX `idx_fpds_idv_piid` (`idv_piid`);
ALTER TABLE `sam_exclusion` ADD UNIQUE INDEX `uk_excl_key` (`uei`, `exclusion_type`, `activation_date`);
ALTER TABLE `usaspending_award` ADD INDEX `idx_usa_modified` (`last_modified_date`);
```

## Finding 3: Column Type Mismatches

| Table | Column | Live DB | DDL (Correct) | Introduced | Risk |
|-------|--------|---------|---------------|------------|------|
| `opportunity_poc` | `notice_id` | `VARCHAR(50)` | `VARCHAR(100)` | Phase 14.14 / 14.24 consolidation. Phase 9 plan specified VARCHAR(100) but live DB was created with VARCHAR(50). | **DATA TRUNCATION** — notice_ids > 50 chars fail on insert. POC loading (Phase 44.3) will hit this once data flows. | `usaspending_award` | `record_hash` | `VARCHAR(64)` | `CHAR(64)` | Phase 14.14 | Low — functional but inconsistent with all other hash columns (CHAR(64)). |

### Fix SQL
```sql

ALTER TABLE `usaspending_award` MODIFY COLUMN `record_hash` CHAR(64);
```

## Finding 4: Index Definition Mismatch — Phase 14.14 (CRITICAL)

| Aspect | Live DB | DDL (Correct) |
|--------|---------|---------------|
| `uk_prospect_notice` | `UNIQUE (notice_id)` | `UNIQUE (organization_id, notice_id)` |

**Origin**: Multi-tenancy was introduced in **Phase 14.5** (completed 2026-03-02). That phase added `organization_id` to `prospect` and the FK, but did not update the unique key. **Phase 14.14** (completed 2026-03-03, one day later) identified this gap and fixed the DDL, but the ALTER TABLE was never applied.

**Impact**: This is the most severe drift item. With `UNIQUE(notice_id)`, only one organization in the entire system can prospect any given opportunity. `ProspectService.cs` (line 54) has application-level protection scoped to `organization_id`, but the DB constraint is stricter — it blocks across organizations. Currently masked because only one organization exists. The moment a second org is onboarded, prospect creation fails for any opportunity already tracked.

### Fix SQL
```sql
-- Verify no duplicate (organization_id, notice_id) pairs exist first:
-- SELECT organization_id, notice_id, COUNT(*) FROM prospect GROUP BY 1,2 HAVING COUNT(*) > 1;
ALTER TABLE `prospect` DROP INDEX `uk_prospect_notice`;
ALTER TABLE `prospect` ADD UNIQUE INDEX `uk_prospect_notice` (`organization_id`, `notice_id`);
```

## Finding 5: Unapplied EF Core Migrations

Two migrations exist in `api/src/FedProspector.Infrastructure/Migrations/` but are NOT in `__efmigrationshistory`. Since we are still building v1, EF Core migrations add unnecessary complexity. These table/column definitions should be consolidated into the Python DDL scripts (the single source of truth).

### Migration 1: `UpdateGsaLaborRateColumns` (20260304061011)
- **Drops**: `opportunity.description`, `gsa_labor_rate.hourly_rate_year1`
- **Adds**: `opportunity.description_url`, `gsa_labor_rate.{category, idv_piid, subcategory}`
- **Note**: Live DB already has these changes (applied via Python DDL). This migration is stale — the `__efmigrationshistory` record was never inserted.

### Migration 2: `AddCompanyProfile` (20260305203312)
- **Adds 19 columns to `organization`**: company profile fields (address, UEI, CAGE, revenue, etc.)
- **Creates 3 new tables**: `organization_certification`, `organization_naics`, `organization_past_performance`
- **Note**: These do NOT exist in the live DB or in any Python DDL file. Needed for Company Profile feature.

### Recommended v1 Approach
1. Add the `AddCompanyProfile` tables/columns to Python DDL (`tables/60_prospecting.sql` or `tables/90_web_api.sql`)
2. Delete EF Core migration files — no migrations needed for v1
3. Insert a dummy `__efmigrationshistory` record if EF Core requires it, or configure EF Core to skip migration checks
