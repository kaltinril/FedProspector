## Overview

The application will be a paid SaaS product where companies subscribe so their employees can prospect federal contracts. This phase retrofits the existing single-tenant auth system into a multi-tenant model and addresses security findings from the codebase audit.

**Multi-tenancy model**: Shared public data (opportunities, awards, entities from SAM.gov) + isolated capture data (prospects, proposals, notes, saved searches, notifications) per organization.

**Initial deployment**: One company (wife's company) to vet and prove the product before expanding.

## Authentication Changes

### 14.5.1 httpOnly Cookie Auth (Replace localStorage)

**Current**: Login returns JWT in JSON body ‚Üí frontend stores in localStorage ‚Üí Axios attaches Bearer header.

**New**: Login sets JWT as an httpOnly cookie ‚Üí browser auto-sends on every request ‚Üí no client-side token access.

**Backend changes**:
- `POST /auth/login`: On success, set `Set-Cookie: access_token=<jwt>; HttpOnly; Secure; SameSite=Strict; Path=/api`
- `POST /auth/refresh`: On success, set new access_token cookie + new refresh_token cookie
- `POST /auth/logout`: Clear both cookies (`Set-Cookie: access_token=; Max-Age=0` and `Set-Cookie: refresh_token=; Max-Age=0`)
- JWT validation: Read token from cookie instead of (or in addition to) Authorization header
- Add CSRF protection: Double-submit cookie pattern (see CSRF specification below)

**Cookie `Domain` note**: Domain attribute intentionally omitted ‚Äî API and UI served same-origin behind reverse proxy. Document explicitly if subdomains are introduced.

**`SameSite=Lax` consideration**: Consider `SameSite=Lax` for the access token cookie if notification emails link directly to the app (Strict blocks cookies on top-level navigations from external sites). Lax still prevents cross-site POST attacks. The CSRF double-submit pattern provides mutation protection regardless.

**CSRF specification**:
- CSRF cookie name: `XSRF-TOKEN` (non-httpOnly, so JavaScript can read it)
- Set on login response and refreshed on each successful auth response
- Contains a cryptographically random value tied to the session
- Server-side middleware validates `X-XSRF-TOKEN` request header matches the `XSRF-TOKEN` cookie value on `POST`, `PATCH`, and `DELETE` requests
- `GET` requests are exempt from CSRF validation
- Non-browser clients using `Authorization: Bearer` header skip CSRF entirely (no cookie = no CSRF needed)

**Frontend changes** (impacts Phase 15):
- No localStorage token management
- Axios: `withCredentials: true` on all requests
- CSRF: Read XSRF token from non-httpOnly cookie, attach as header
- Auth state: Call `GET /auth/me` on app load to check if session is active

### 14.5.2 Token Refresh Flow (Opaque Refresh Token)

**Design**: Separate opaque refresh token stored in a second httpOnly cookie, independent of the JWT access token.

**Refresh token details**:
- Cryptographically random 256-bit value generated via `RandomNumberGenerator`
- Stored **hashed** (SHA-256) in `app_session` table (never store plaintext)
- Delivered as cookie: `refresh_token; HttpOnly; Secure; SameSite=Lax; Path=/api/v1/auth/refresh`
- Access token cookie remains: `access_token; HttpOnly; Secure; SameSite=Strict; Path=/api`

**Token lifetimes**:
- Access token: 30 minutes (reduced from 24 hours)
- Refresh token: 7 days (configurable)

**Refresh token rotation**:
- Each call to `POST /api/v1/auth/refresh` issues **both** a new access token and a new refresh token
- The old refresh token hash is invalidated in `app_session` immediately
- If an already-rotated (invalidated) refresh token is reused, **revoke ALL sessions for that user** ‚Äî this indicates token theft (attacker and legitimate user both have a copy)

**Endpoint**: `POST /api/v1/auth/refresh`
- Reads the `refresh_token` cookie
- Looks up the SHA-256 hash in `app_session`
- If found and not revoked: issue new access + refresh tokens, invalidate old refresh token
- If not found or revoked: return 401, clear both cookies
- If reuse detected (token was already rotated): revoke all user sessions, return 401, log security event

**Frontend** (Phase 15): Axios 401 interceptor attempts silent refresh before redirecting to login.

### 14.5.3 Session Validation (OnTokenValidated)

Add `JwtBearerEvents.OnTokenValidated` handler:
1. Extract token hash from cookie
2. Check `app_session` table: session exists AND `revoked_at IS NULL`
3. Reject if session is revoked (logout, password change, admin deactivation)
4. Cache result in `IMemoryCache` (**30-second TTL**) to avoid DB hit on every request

**Cache invalidation for critical revocations**: Password changes and admin deactivation add an entry to a short-lived in-memory revocation list. The `OnTokenValidated` handler checks this list first (before the positive cache), ensuring immediate revocation even if the session was recently cached as valid.

**Multi-instance note**: For multi-instance deployments, migrate from `IMemoryCache` to `IDistributedCache` (Redis) so that revocations propagate across all instances.

This makes logout and password change **actually revoke** sessions immediately.

### 14.5.4 Invite-Only Registration

**Remove**: Open `POST /auth/register` endpoint.

**New flow**:
1. Company admin calls `POST /api/v1/org/invites` with email + role
2. System generates invite code, stores in `organization_invite`, sends email (or returns code for manual sharing)
3. Invitee calls `POST /api/v1/auth/register` with invite code + password + display name
4. System validates invite code, creates user in the admin's organization, marks invite accepted

**Super-admin** can create organizations and the first owner user directly.

**Invite token security**:
- Invite token: 32 bytes (256 bits) generated from `RandomNumberGenerator`
- Default invite expiry: 72 hours
- Rate limit on register endpoint: 3 attempts per minute per IP
- Lock invite code after 5 failed validation attempts (set `locked_at` timestamp)
- Log all failed invite code attempts to `activity_log` with source IP

**MVP note**: Invite codes are shared manually (no email infrastructure). Email delivery (SendGrid/SMTP) required before onboarding second customer.

**Known gap**: Self-service password reset (forgot password) deferred ‚Äî requires email infrastructure. MVP uses admin-initiated reset via `POST /admin/users/{id}/reset-password`.

## Security Fixes

### 14.5.5 Security Header Hardening

Add to `SecurityHeadersMiddleware.cs`:
```
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; font-src 'self'; connect-src 'self'; frame-ancestors 'none';
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

### 14.5.6 Swagger Restriction

Restrict Swagger UI to Development environment only:
```csharp
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI(...);
}
```

### 14.5.6a Additional Security Fixes

- **Account enumeration prevention**: Return "Invalid email or password" for ALL login failure cases (locked account, inactive account, wrong password, unknown user). Never reveal which condition failed.
- **Email change requires current password**: Prevent session-hijack account takeover by requiring current password verification in `UpdateUserAsync` when email is being changed.
- **Admin audit trail**: Add `_activityLogService.LogAsync()` calls in `UpdateUserAsync` and `ResetPasswordAsync` for complete admin audit trail.
- **Role consolidation**: System-level role derives from `IsAdmin` flag. Organization-level role derives from `org_role` column. Remove any redundant `Role` field to maintain a single authoritative source per level.
- **Connection string security**: Move database connection string to environment variables. Committed `appsettings.json` must contain a placeholder only (e.g., `"ConnectionString": "SET_VIA_ENVIRONMENT"`). Production `appsettings.json` must use `SslMode=Required` for MySQL TLS.
- **Force password change**: Add `force_password_change` column (`CHAR(1) DEFAULT 'N'`) to `app_user`. Set to `'Y'` after admin-initiated password reset. Enforce via middleware ‚Äî if flag is set, all requests except `POST /auth/change-password` return 403 with message "Password change required."

### 14.5.7 CORS Tightening

Replace `AllowAnyHeader()` / `AllowAnyMethod()` with explicit lists:
- Headers: `Authorization`, `Content-Type`, `Accept`, `X-XSRF-TOKEN`
- Methods: `GET`, `POST`, `PATCH`, `DELETE`, `OPTIONS`

### 14.5.8 Rate Limiter Improvements

- Add `Retry-After` header to 429 responses
- Add `retryAfterSeconds` field to the JSON response body
- **Global login rate limit**: 100 total login attempts per minute across all IPs (prevents distributed brute force)
- **Progressive delay**: After 3 failures for a given email within 10 minutes, add 2-second artificial delay to login responses
- **Lockout response**: Use identical message as invalid credentials ("Invalid email or password") to prevent account enumeration

### 14.5.8a HTTPS Redirect Fix

Gate `UseHttpsRedirection()` behind `!app.Environment.IsDevelopment()` ‚Äî HTTPS redirect breaks Vite dev proxy which targets HTTP.

### 14.5.8b CORS Fallback Fix

Change CORS fallback default from `localhost:3000` to `localhost:5173` (Vite default dev server port).

### 14.5.9 JWT Clock Skew

Set `ClockSkew = TimeSpan.Zero` in `TokenValidationParameters`.

### 14.5.10 Generic Error Messages

Replace internal exception details in HTTP responses with generic messages:
- `KeyNotFoundException` ‚Üí "The requested resource was not found."
- `InvalidOperationException` ‚Üí "The request could not be processed."
- Log detailed messages server-side (already happening).

## Known Gap: Email Infrastructure

**Status**: Not in scope for Phase 14.5. Required before onboarding a second customer.

MVP shares invite codes manually (admin gives code to invitee via Slack, email, etc.). This works for the initial deployment with one company (wife's company) but does not scale.

**Before production launch with multiple customers, add email infrastructure for**:
- Invite delivery (new team member receives invite link via email)
- Password reset links (self-service forgot-password flow)
- Notification digests (daily/weekly summary of new matching opportunities)

**Recommended approach**: SendGrid (free tier: 100 emails/day) or AWS SES. Integrate as a shared service callable from both the C# API and the Python scheduler.

**Impact if deferred**: Each new customer requires manual invite code sharing. No self-service password reset (admin must do it). No email-based notifications.
