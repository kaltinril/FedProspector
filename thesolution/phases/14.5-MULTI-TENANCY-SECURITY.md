## Overview

The application will be a paid SaaS product where companies subscribe so their employees can prospect federal contracts. This phase retrofits the existing single-tenant auth system into a multi-tenant model and addresses security findings from the codebase audit.

**Multi-tenancy model**: Shared public data (opportunities, awards, entities from SAM.gov) + isolated capture data (prospects, proposals, notes, saved searches, notifications) per organization.

**Initial deployment**: One company (wife's company) to vet and prove the product before expanding.

## Data Model Changes

### New Table: `organization`

```sql
CREATE TABLE organization (
    organization_id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(200) NOT NULL,
    slug VARCHAR(100) NOT NULL UNIQUE,        -- URL-friendly identifier
    is_active CHAR(1) NOT NULL DEFAULT 'Y',
    max_users INT NOT NULL DEFAULT 10,        -- seat limit (for future billing)
    subscription_tier VARCHAR(50) DEFAULT 'trial',  -- trial, standard, premium (future)
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### Modified Table: `app_user`

Add columns:
- `organization_id INT NOT NULL` (FK → organization)
- `org_role VARCHAR(50) NOT NULL DEFAULT 'member'` — values: `owner`, `admin`, `member`
- `invited_by INT NULL` (FK → app_user)
- `invite_accepted_at DATETIME NULL`

The existing `role` column (`admin`/empty) becomes the **system-level** role (super-admin for platform management). The new `org_role` is the **company-level** role.

### New Table: `organization_invite`

```sql
CREATE TABLE organization_invite (
    invite_id INT AUTO_INCREMENT PRIMARY KEY,
    organization_id INT NOT NULL,
    email VARCHAR(255) NOT NULL,
    invite_code VARCHAR(64) NOT NULL UNIQUE,  -- SHA-256 of random token
    org_role VARCHAR(50) NOT NULL DEFAULT 'member',
    invited_by INT NOT NULL,                  -- FK → app_user
    expires_at DATETIME NOT NULL,
    accepted_at DATETIME NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (organization_id) REFERENCES organization(organization_id),
    FOREIGN KEY (invited_by) REFERENCES app_user(user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## Authentication Changes

### 14.5.1 httpOnly Cookie Auth (Replace localStorage)

**Current**: Login returns JWT in JSON body → frontend stores in localStorage → Axios attaches Bearer header.

**New**: Login sets JWT as an httpOnly cookie → browser auto-sends on every request → no client-side token access.

**Backend changes**:
- `POST /auth/login`: On success, set `Set-Cookie: access_token=<jwt>; HttpOnly; Secure; SameSite=Strict; Path=/api`
- `POST /auth/refresh`: On success, set new access_token cookie
- `POST /auth/logout`: Clear the cookie (`Set-Cookie: access_token=; Max-Age=0`)
- JWT validation: Read token from cookie instead of (or in addition to) Authorization header
- Add CSRF protection: Double-submit cookie pattern (`X-XSRF-TOKEN` header)

**Frontend changes** (impacts Phase 15):
- No localStorage token management
- Axios: `withCredentials: true` on all requests
- CSRF: Read XSRF token from non-httpOnly cookie, attach as header
- Auth state: Call `GET /auth/me` on app load to check if session is active

### 14.5.2 Token Refresh Flow

**New endpoint**: `POST /api/v1/auth/refresh`
- Reads the current access_token cookie
- If token is expired but within a refresh window (e.g., expired < 7 days ago), issue a new token
- Validates session is still active in `app_session` table
- Returns new access_token cookie
- If refresh fails → 401 → frontend redirects to login

**Access token lifetime**: Reduce from 24 hours to 30 minutes.
**Refresh window**: 7 days from original login (configurable).

**Frontend** (Phase 15): Axios 401 interceptor attempts silent refresh before redirecting to login.

### 14.5.3 Session Validation (OnTokenValidated)

Add `JwtBearerEvents.OnTokenValidated` handler:
1. Extract token hash from cookie
2. Check `app_session` table: session exists AND `revoked_at IS NULL`
3. Reject if session is revoked (logout, password change, admin deactivation)
4. Cache result in `IMemoryCache` (1-2 minute TTL) to avoid DB hit on every request

This makes logout and password change **actually revoke** sessions immediately.

### 14.5.4 Invite-Only Registration

**Remove**: Open `POST /auth/register` endpoint.

**New flow**:
1. Company admin calls `POST /api/v1/org/invites` with email + role
2. System generates invite code, stores in `organization_invite`, sends email (or returns code for manual sharing)
3. Invitee calls `POST /api/v1/auth/register` with invite code + password + display name
4. System validates invite code, creates user in the admin's organization, marks invite accepted

**Super-admin** can create organizations and the first owner user directly.

## Security Fixes

### 14.5.5 Security Header Hardening

Add to `SecurityHeadersMiddleware.cs`:
```
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; font-src 'self'; connect-src 'self'; frame-ancestors 'none';
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

### 14.5.6 Swagger Restriction

Restrict Swagger UI to Development environment only:
```csharp
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI(...);
}
```

### 14.5.7 CORS Tightening

Replace `AllowAnyHeader()` / `AllowAnyMethod()` with explicit lists:
- Headers: `Authorization`, `Content-Type`, `Accept`, `X-XSRF-TOKEN`
- Methods: `GET`, `POST`, `PATCH`, `DELETE`, `OPTIONS`

### 14.5.8 Rate Limiter Improvements

- Add `Retry-After` header to 429 responses
- Add `retryAfterSeconds` field to the JSON response body

### 14.5.9 JWT Clock Skew

Set `ClockSkew = TimeSpan.Zero` in `TokenValidationParameters`.

### 14.5.10 Generic Error Messages

Replace internal exception details in HTTP responses with generic messages:
- `KeyNotFoundException` → "The requested resource was not found."
- `InvalidOperationException` → "The request could not be processed."
- Log detailed messages server-side (already happening).

## Role Hierarchy

```
Super Admin (system role)
  └── Can create/manage organizations, view all data, manage platform

Organization Owner (org role)
  └── Can manage org settings, invite/remove members, promote to admin

Organization Admin (org role)
  └── Can invite members, manage team, view all org prospects

Organization Member (org role)
  └── Can search, create prospects, manage their own assignments
```

**JWT claims** after this phase:
- `sub`: user ID
- `email`: user email
- `role`: system role (`admin` or empty)
- `org_id`: organization ID
- `org_role`: organization role (`owner`, `admin`, `member`)
