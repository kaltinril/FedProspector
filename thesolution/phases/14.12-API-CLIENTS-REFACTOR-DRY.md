# Phase 14.12 — API Client DRY Refactor

## Status: COMPLETE (2026-03-03)

---

## Goal

Code review of `fed_prospector/api_clients/` found five patterns duplicated across 4-5 client files that are large enough to be worth extracting. This phase removes the duplication. Focus is on real wins — not every 3-line similarity.

**Out of scope deliberately:**
- Single-record lookup abstraction (`get_organization`, `check_entity`, etc.) — 60-80 lines total but each lookup has unique response parsing and error handling; generalizing adds complexity, not removes it
- Convenience wrapper methods (`search_by_naics`, `search_by_prime`, etc.) — thin wrappers that delegate to `_all` methods; acceptable as-is
- Logging consolidation — every pagination loop logs slightly differently, not worth normalizing
- `SAMOpportunityClient` pagination — already uses a date-chunking strategy unique to that API; leave it alone

---

## The Five Wins

### Win 1 — Parameterize `BaseAPIClient.paginate()`: Eliminate ~220 Lines of Custom Pagination

**The problem:** `BaseAPIClient.paginate()` (lines 193-220 of `base_client.py`) handles only offset-based pagination with hardcoded key names. Five clients each implement their own 35-45 line `search_X_all()` pagination loop because the base method is too rigid:

| Client | Style | Results key | End-of-page detection |
|--------|-------|-------------|----------------------|
| `SAMAwardsClient.search_awards_all()` | offset-as-page-index | `awardSummary` | `page * limit >= totalRecords` |
| `SAMExclusionsClient.search_exclusions_all()` | page-based (0-indexed) | `excludedEntity` | `page * size >= totalRecords` |
| `SAMSubawardClient.search_subcontracts_all()` | page-based (0-indexed) | `data` | `page_number >= totalPages` |
| `USASpendingClient.search_awards_all()` | page-based (1-indexed) | `results` | `page_metadata.hasNext == False` |
| `SAMFedHierClient.search_organizations_all()` | offset-based | `orglist` | `offset >= totalrecords` (lowercase key) |

**Important quirk — SAMAwards offset:** The SAM Contract Awards API uses the `offset` parameter as a **page index**, not a record offset. Page 0 returns records 0-99, page 1 returns records 100-199, etc. This differs from SAMFedHier where `offset` truly is a record offset.

**The fix:** Replace the existing `paginate()` in `BaseAPIClient` with a parameterized version:

```python
def paginate(self, endpoint, params, page_size=100,
             pagination_style="offset",    # "offset" | "page"
             offset_param="offset",        # query param name for offset
             page_param="page",            # query param name for page number
             size_param="limit",           # query param name for page size
             page_start=0,                 # first page index (0 or 1)
             total_key="totalRecords",     # response key for total count
             results_key="data",           # response key for record list
             has_next_key=None,            # if set, use response[has_next_key] to stop
             total_pages_key=None):        # if set, use response[total_pages_key] to stop
    """Generic paginator — parameterize instead of reimplementing.

    Supports three end-of-page detection strategies:
      1. has_next_key  — stop when response[has_next_key] is falsy (USASpending)
      2. total_pages_key — stop when page >= response[total_pages_key] (Subaward)
      3. total_key — stop when offset/page*size >= response[total_key] (SAM default)

    Yields the parsed results list for each page (not the full response).
    """
    if params is None:
        params = {}
    params[size_param] = page_size

    page = page_start
    offset = 0

    while True:
        if pagination_style == "page":
            params[page_param] = page
        else:  # offset
            params[offset_param] = offset

        response = self.get(endpoint, params=params)
        data = response.json()
        results = data.get(results_key, [])

        self.logger.info(
            "Page %s: %d results", page if pagination_style == "page" else f"offset={offset}",
            len(results),
        )

        yield results

        if not results:
            break

        # Detect end of results
        if has_next_key is not None:
            if not data.get(has_next_key, False):
                break
            page += 1
        elif total_pages_key is not None:
            total_pages = data.get(total_pages_key, 0)
            page += 1
            if page >= total_pages:
                break
        else:
            total = data.get(total_key, 0)
            if pagination_style == "page":
                page += 1
                if page * page_size >= total:
                    break
            else:
                offset += page_size
                if offset >= total:
                    break
```

Each client's `search_X_all()` method then becomes:


**Estimated savings:** ~170 lines across 4 SAM clients.

### Win 2 — `_format_date()` in `BaseAPIClient`: Eliminate ~100 Lines of Per-Client Date Formatting

**The problem:** 5 clients define identical or near-identical `_format_date` `@staticmethod` methods. The logic is the same — convert `date`/`datetime`/`string` to a formatted string. Only the output format differs:

| Client | Method | Output format | Call sites |
|--------|--------|--------------|-----------|
| `SAMSubawardClient` | `_format_date(d)` | `%Y-%m-%d` | 2 |
| `SAMFedHierClient` | `_format_date(d)` | `%Y-%m-%d` | 2 |
| `USASpendingClient` | `_format_date(value)` | `%Y-%m-%d` | 4 |
| `SAMEntityClient` | `_format_date(value)` | `%m/%d/%Y` | 1 |
| `SAMOpportunityClient` | `_format_date(value)` | `%m/%d/%Y` | 4 |

`SAMAwardsClient` is different — it defines `_format_date_range()` which wraps two dates into SAM's `[MM/DD/YYYY,MM/DD/YYYY]` bracket syntax. That method stays per-client because the bracket syntax is unique to the Awards API.

**The fix:** Add `_format_date()` as an instance method (not `@staticmethod`) to `BaseAPIClient`:

```python
def _format_date(self, value, fmt="%Y-%m-%d"):
    """Convert date, datetime, or string to formatted string.

    Args:
        value: A date, datetime, or string. None returns None.
        fmt: strftime format string. Default: '%Y-%m-%d'.

    Returns:
        str: Formatted date string, or None if value is None.
    """
    if value is None:
        return None
    if isinstance(value, (date, datetime)):
        return value.strftime(fmt)
    return str(value)
```

Each of the 5 clients removes its `_format_date` `@staticmethod`. The 3 clients that use `%Y-%m-%d` (SAMSubaward, SAMFedHier, USASpending) call `self._format_date(v)` with no format change. The 2 clients that use `%m/%d/%Y` (SAMEntity, SAMOpportunity) update their call sites to `self._format_date(v, "%m/%d/%Y")`.

All call sites already use `self._format_date(...)` — no refactoring of callers beyond adding the `fmt` argument where needed.

**Why instance method not staticmethod:** The base class already uses instance methods for all helpers. Making it a staticmethod would be inconsistent and prevents subclasses from overriding behavior (e.g., if a future API needed a different fallback strategy).

**Estimated savings:** ~100 lines (5 clients × ~20 lines each).

### Win 3 — `_sam_init_kwargs()` Class Method: Eliminate ~72 Lines of Copied `__init__` Code

**The problem:** 4 clients (SAMAwards, SAMExclusions, SAMFedHier, SAMSubaward) have identical `__init__` methods that differ only in the `source_name` string:

```python
# Repeated verbatim in 4 files, only "SAM_AWARDS" / "SAM_EXCLUSIONS" etc. changes
def __init__(self, api_key_number=1):
    if api_key_number == 2:
        api_key = settings.SAM_API_KEY_2
        daily_limit = settings.SAM_DAILY_LIMIT_2
    else:
        api_key = settings.SAM_API_KEY
        daily_limit = settings.SAM_DAILY_LIMIT
    source_name = "SAM_AWARDS"   # ← only this line differs
    super().__init__(
        base_url=settings.SAM_API_BASE_URL,
        api_key=api_key,
        source_name=source_name,
        max_daily_requests=daily_limit,
    )
```

**The fix:** Add a `@classmethod` to `BaseAPIClient`:

```python
@classmethod
def _sam_init_kwargs(cls, source_name, api_key_number=1):
    """Return kwargs dict for super().__init__() using SAM's dual-key config.

    Centralizes the SAM_API_KEY / SAM_API_KEY_2 selection logic shared by
    all four SAM search clients (Awards, Exclusions, FedHier, Subaward).

    Args:
        source_name: Source system name for rate tracking (e.g. "SAM_AWARDS").
        api_key_number: 1 (default, 10/day) or 2 (1000/day).

    Returns:
        dict: kwargs suitable for BaseAPIClient.__init__().
    """
    from config import settings
    if api_key_number == 2:
        api_key = settings.SAM_API_KEY_2
        limit = settings.SAM_DAILY_LIMIT_2
    else:
        api_key = settings.SAM_API_KEY
        limit = settings.SAM_DAILY_LIMIT
    return dict(
        base_url=settings.SAM_API_BASE_URL,
        api_key=api_key,
        source_name=source_name,
        max_daily_requests=limit,
    )
```

Each client's `__init__` shrinks to 2 lines:

```python
def __init__(self, api_key_number=1):
    super().__init__(**self._sam_init_kwargs("SAM_AWARDS", api_key_number))
```

The docstring and `Args` block for `api_key_number` moves to `_sam_init_kwargs` — each client's `__init__` keeps a one-line docstring referencing it.

`SAMExtractClient.__init__` is not identical (it accepts `api_key=None` directly, not `api_key_number`) — leave it unchanged.

**Estimated savings:** ~72 lines (~18 per client × 4 clients).

### Win 4 — Fix `BaseAPIClient.get()` to Skip Empty `api_key`: Eliminate 2 `get()` Overrides

**The problem:** `BaseAPIClient.get()` unconditionally adds `params["api_key"] = self.api_key`. When `api_key=""` (both `USASpendingClient` and `CalcPlusClient` pass `api_key=""`), this appends an empty `api_key=` parameter to every request. Both clients work around this with identical 8-line `get()` overrides:

```python
# USASpendingClient (lines 477-480) — identical logic to CalcPlusClient (lines 103-113)
def get(self, endpoint, params=None, **kwargs):
    """Override base get() to avoid injecting an api_key parameter."""
    url = f"{self.base_url}{endpoint}"
    return self._request_with_retry("GET", url, params=params, **kwargs)
```

**The fix:** One guard in `BaseAPIClient.get()`:

```python
def get(self, endpoint, params=None, **kwargs):
    """Convenience method for GET requests."""
    url = f"{self.base_url}{endpoint}"
    if params is None:
        params = {}
    if self.api_key:   # Only add if non-empty string
        params["api_key"] = self.api_key
    return self._request_with_retry("GET", url, params=params, **kwargs)
```

Then delete the `get()` override from both `USASpendingClient` and `CalcPlusClient` entirely.

**Estimated savings:** ~20 lines (2 override methods removed).

### Win 5 — `get_binary()` in `BaseAPIClient`: Eliminate Manual Rate-Check in `SAMExtractClient`

**The problem:** `SAMExtractClient._download_extract_file()` (lines 242-321) bypasses `_request_with_retry()` entirely. It manually:
1. Calls `_check_rate_limit()` directly (duplicating the guard in `_request_with_retry`)
2. Uses `self.session.get()` directly to get a streaming response
3. Calls `_increment_rate_counter()` manually after the request

This works but leaves the rate-limit guard fragile — if the base class logic changes, `SAMExtractClient` won't pick it up.

**The fix:** Add `get_binary()` to `BaseAPIClient` that respects rate limits and retry infrastructure but returns the raw streaming `Response` object instead of parsed JSON:

```python
def get_binary(self, endpoint, params=None, stream=True, **kwargs):
    """Like get() but returns the raw Response for binary/streaming downloads.

    Handles rate-limit checking and retry via _request_with_retry().
    The caller is responsible for reading and closing the response.

    Args:
        endpoint: URL path relative to base_url.
        params: Query parameters dict. api_key is added automatically if set.
        stream: If True, the response body is not downloaded immediately
                (use response.iter_content() to stream). Default True.

    Returns:
        requests.Response: Raw response object.

    Raises:
        RateLimitExceeded: If daily limit has been reached.
        requests.HTTPError: On non-200 HTTP responses after all retries.
    """
    url = f"{self.base_url}{endpoint}"
    if params is None:
        params = {}
    if self.api_key:
        params["api_key"] = self.api_key
    return self._request_with_retry("GET", url, params=params, stream=stream, **kwargs)
```

`SAMExtractClient._download_extract_file()` then replaces its manual rate check + `session.get()` block with:

```python
response = self.get_binary(
    self.EXTRACT_ENDPOINT,
    params={"fileName": filename},
    stream=True,
    timeout=60,
)
```

The 404 check and file-streaming-to-disk logic stays in `_download_extract_file()` — that is specific to this client and belongs there.

**Note on retry behavior:** `_request_with_retry()` currently retries on 429 and 5xx but not 404. For extract downloads, 404 means "file doesn't exist for this date" — a normal condition, not an error. The current code handles this explicitly with `if response.status_code == 404: return None`. After the refactor, `_request_with_retry` will call `response.raise_for_status()` on 4xx (not 429), which will raise `HTTPError` for 404. The `_download_extract_file()` method must catch `HTTPError` with status 404 and return `None` instead, or `_request_with_retry` needs a `skip_raise_for` parameter. **Decision: catch `HTTPError` for 404 in `_download_extract_file()`** — it keeps the base class simple.

**Estimated savings:** ~40 lines (the manual rate-check + session.get block in `_download_extract_file`).

## Bug Fixes (from Phase 14.x Code Review)

Code review of `fed_prospector/api_clients/` found five bugs independent of the DRY work above. Fix these in the same PR — they are small and several overlap with Win 1/5 touch points.

### CRITICAL

**Rate counter incremented on every retry, not just the final call** (`base_client.py:126-133`)

`_increment_rate_counter()` is called unconditionally after every HTTP attempt, including retried 429 and 5xx responses. A single request that triggers 3 retries before succeeding consumes 4 quota units instead of 1. On the 10-calls/day free key, a single transient 503 on the first call of the day can exhaust the entire daily budget for only 4 logical interactions. Fix: move `_increment_rate_counter()` so it is called only on the final successful response (status 200), or restructure to count logical calls rather than HTTP attempts.

---

### HIGH

**`paginate()` mutates the caller's `params` dict in-place** (`base_client.py:204-206`)

`paginate()` writes `offset` and `limit` keys directly into the dict passed by the caller. If the caller reuses the same dict across calls (e.g. in a loop), the offset from the previous page leaks into the next invocation. Fix: copy `params` at the top of `paginate()` before modifying it — `params = dict(params or {})`.

**`_parse_date` crashes on `YYYY-MM-DD` input** (`sam_opportunity_client.py:415-430`)

The method only handles `MM/dd/yyyy` format. Any `YYYY-MM-DD` string — e.g. passed by a downstream caller or future ETL change — raises `ValueError` with a confusing message. The ETL-layer `parse_date()` in `etl/etl_utils.py` already handles both formats. Fix: use `datetime.strptime` with a try for both `%m/%d/%Y` and `%Y-%m-%d`, or delegate to `etl.etl_utils.parse_date`.

---

### MEDIUM

**Budget enforcement ignores pagination calls** (`sam_opportunity_client.py:379-381`)

`calls_used` is incremented by `len(date_chunks)`, assuming one API call per chunk. If any chunk triggers multiple pages via `paginate()`, those extra calls are not counted. A 5-call budget can be silently exceeded on large date ranges. Fix: track actual calls made by passing a mutable counter into `paginate()`, or check `self.get_rate_remaining()` before each page fetch.

**API key validation uses a hardcoded historical date range** (`setup.py:244-251`)

`_check_sam_api_key()` builds a test URL with hardcoded dates `"&postedFrom=01/01/2026&postedTo=01/02/2026"`. As time passes this becomes a stale historical query. Additionally, the API key is appended to the URL directly, bypassing `_increment_rate_counter()`, so the reported "remaining calls" is 1 too high after validation runs. Fix: compute the date range dynamically as `today - 1 day` to `today`, and route the request through the normal `BaseAPIClient.get()` path so rate counting stays consistent.

## Implementation Order

1. **Fix `BaseAPIClient.get()` for empty api_key** — one-line change, remove 2 overrides, test immediately. Zero risk.
2. **Add `_sam_init_kwargs()` classmethod** — migrate 4 `__init__` methods one file at a time. Quick win.
3. **Add `_format_date()` instance method** — remove 5 per-client staticmethods; update `%m/%d/%Y` call sites in SAMEntity and SAMOpportunity. Quick win.
4. **Add `get_binary()` and migrate `SAMExtractClient`** — refactor `_download_extract_file()`. Test with a known-good extract filename.
5. **Extend `paginate()`** — add parameters to the base method (backward-compatible since all new params have defaults). Test the existing `SAMOpportunityClient` still works.
6. **Migrate pagination loops one client at a time** — SAMFedHier → SAMExclusions → SAMSubaward → SAMAwards. Test each client after migration.

## Risks

| Risk | Mitigation |
|------|-----------|
| `paginate()` parameter explosion — 10 params is a lot for a base class method | All parameters have sensible defaults; the common case (offset-based SAM) uses 2-3 params. Document each group clearly. |
| SAMAwards `offset` is a page index, not record offset — easy to confuse | The `pagination_style="page"` + `page_param="offset"` combination handles it; add a code comment at the call site explaining the API quirk |
| `SAMFedHierClient` uses `totalrecords` (lowercase) vs every other client's `totalRecords` | Pass `total_key="totalrecords"` explicitly; add a comment noting this is a SAM API inconsistency |
| `get_binary()` + `_request_with_retry()` changes 404 from silent `return None` to `raise HTTPError` | Catch `requests.HTTPError` for 404 in `_download_extract_file()` and return None; document this explicitly |
| USASpending and CALC+ `get()` overrides are removed — if a new method depends on the override being present, it breaks silently | Both overrides only skipped api_key injection; the fix in `get()` itself is equivalent. No behavior change for these clients. |
| Backward compatibility of `paginate()` — `SAMOpportunityClient` calls `self.paginate(...)` with the old 3-param signature | New signature is fully backward-compatible: `offset_key` is renamed to `offset_param` — verify `SAMOpportunityClient` passes kwarg by name or update the call site |
