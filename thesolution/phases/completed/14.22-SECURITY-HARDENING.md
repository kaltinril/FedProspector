Each finding includes a **Verdict** that answers three questions:
1. **Real concern or theater?** — Would this actually hurt us, or is it theoretical noise?
2. **Maintainable?** — Does the fix keep the codebase simple and clean?
3. **Action** — FIX, ACCEPT (document risk), or SKIP (security theater / not worth the complexity)

## Findings — CRITICAL

### WHR-001: Temporary Password Logged at DEBUG Level [CRITICAL]
- **Category**: Data Leak | **CWE**: CWE-532 | **OWASP**: A09:2021
- **Location**: `api/src/FedProspector.Infrastructure/Services/AdminService.cs:176`
- **Code**:
  ```csharp
  _logger.LogDebug("Temporary password for user {UserId}: {TempPassword}", userId, tempPassword);
  ```
- **Impact**: Anyone with log access reads plaintext temporary passwords.
- **Fix**: Delete the log line.
- **Verdict**: **FIX** — Real concern. One-line delete. No downside.
- [x] Done

---

### WHR-002: JWT Tokens Returned in HTTP Response Body [CRITICAL]
- **Category**: Data Leak | **CWE**: CWE-200 | **OWASP**: A01:2021
- **Location**: `api/src/FedProspector.Api/Controllers/AuthController.cs:52,113,143`
- **Code**: `return Ok(result);` where `AuthResult` contains `Token` and `RefreshToken`
- **Impact**: Any XSS steals both tokens, defeating httpOnly cookies entirely.
- **Fix**: Return only non-sensitive fields (`UserId`, `UserName`, `ExpiresAt`) when setting cookies. Strip `Token` and `RefreshToken` from the response body.
  ```csharp
  SetAuthCookies(result);
  return Ok(new { result.UserId, result.UserName, result.ExpiresAt });
  ```
- **Verdict**: **FIX** — Real concern. Simple change. The whole point of httpOnly cookies is to prevent JS from reading tokens — returning them in JSON defeats the purpose.
- [x] Done

---

### WHR-003: ZIP Slip Path Traversal in SAM Extract Client [CRITICAL]
- **Category**: Injection | **CWE**: CWE-22 | **OWASP**: A01:2021
- **Location**: `fed_prospector/api_clients/sam_extract_client.py:190-191`
- **Code**: `zf.extractall(output_dir)` — no path validation on ZIP members
- **Impact**: A malicious/tampered ZIP can write files anywhere the process user can write.
- **Fix**: Python 3.12+ one-liner:
  ```python
  zf.extractall(output_dir, filter='data')
  ```
- **Verdict**: **FIX** — Real concern. One parameter change. We're on Python 3.14, so `filter='data'` is available. Source is SAM.gov (trusted), but defense-in-depth costs nothing here.
- [x] Done

## Positive Observations

The audit found strong security fundamentals already in place:

1. **httpOnly cookie auth** with Secure flag and CSRF double-submit protection
2. **Session validation on every request** via `OnTokenValidated` with DB lookup
3. **Refresh token rotation** with reuse detection (steals trigger full session revocation)
4. **BCrypt Enhanced password hashing** (SHA-384 prehash + BCrypt 11 rounds)
5. **Progressive login delays** + account lockout (5 failures → 30 min lock)
6. **Security headers** (X-Content-Type-Options, X-Frame-Options: DENY, CSP, Referrer-Policy, Permissions-Policy, HSTS)
7. **Rate limiting** across all 5 endpoint tiers (auth, login, search, write, admin)
8. **CSV formula injection prevention** (prefixes =, +, -, @ with single quote)
9. **SHA-256 token hashing** in database (can't steal tokens from DB dump)
10. **Invite-only registration** (no open signup)
11. **Parameterized SQL everywhere** in Python ETL (all data values use `%s` placeholders)
12. **Global exception handler** prevents stack traces reaching clients
13. **Activity logging** for all auth events
14. **Subprocess security** — all `subprocess.run()` calls use list-form args, no `shell=True`

## Accepted Risk Documentation

### WHR-009: API Keys in Query Parameters
**Risk**: SAM.gov API keys visible in server/proxy logs.
**Mitigation**: SAM.gov API design constraint — they require `api_key` as query param. Our logging does not log full URLs at INFO level. Keys are free-tier with limited value.

### WHR-010: No MySQL TLS/SSL
**Risk**: DB traffic in plaintext.
**Mitigation**: Localhost-only deployment. No network traversal. Will add SSL when/if deploying to remote DB.

### WHR-012: Temp Password on Console
**Risk**: Temporary password visible on CLI screen.
**Mitigation**: CLI is a trusted operator tool. Admin needs to see the temp password to communicate it. Terminal should be cleared after use.

### WHR-019: Dev DB Credentials in appsettings
**Risk**: Local dev credentials in source control.
**Mitigation**: Local MySQL instance only. Password only useful on localhost. Documented in QUICKSTART.md.

### WHR-023: No Download Integrity Check
**Risk**: Downloaded SAM.gov files not hash-verified.
**Mitigation**: HTTPS provides transport integrity. SAM.gov doesn't publish checksums. ZIP validation via zipfile module. Source is trusted government endpoint.

### WHR-028: Email in Failed Login Logs
**Risk**: PII (email) in activity logs.
**Mitigation**: Required for security monitoring (detecting brute-force targeting). Masking would defeat the audit purpose.

### WHR-033: No HTTPS in Development
**Risk**: Dev traffic unencrypted.
**Mitigation**: Intentional for Vite dev proxy compatibility. Production enables HTTPS redirect.

### WHR-035: In-Memory Rate Limits
**Risk**: Rate limits not shared across instances.
**Mitigation**: Single-instance deployment. Will migrate to Redis when scaling horizontally.
