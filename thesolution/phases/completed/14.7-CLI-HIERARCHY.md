# Phase 14.7: CLI Hierarchy Refactor

**Status**: COMPLETE (2026-03-02)
**Dependencies**: Phase 14.6 (Admin Operability — all 54 commands must exist before renaming)
**Deliverable**: 7-group discoverable CLI replacing 54 flat root commands; `python main.py` shows only groups
**Repository**: `fed_prospector/main.py`

---

> **UX DEBT**: The CLI has 54 flat commands at root level. Users must memorize all names. Opaque names like `calc`, `history`, and `check` are not self-documenting. This phase is a pure UX refactor — zero ETL logic changes.

## Problem

`python main.py` dumps 54 commands in alphabetical order with no grouping. There is no way to discover related commands or understand the system's structure from the help output. Several command names are opaque (`load-calc`, `check`, `history`) or inconsistent (`check-exclusion` is a point lookup; `check-prospects` is a bulk audit). New users have no entry point.

## Goal

Seven top-level groups. `python main.py` shows only the 7 groups. `python main.py load` shows the 10 load sub-commands. Maximum 2 levels deep — no further nesting. All existing command logic is preserved unchanged.

## Implementation Approach

- Pure UX refactor — no ETL logic changes, no moves of command handler code
- Use Click `@group()` nesting: each of the 7 groups is a `@click.group()`, added to the root group via `cli.add_command()`
- Commands renamed for clarity where the old name was opaque or inconsistent
- `HelpGroup` class and `CATEGORIES` dict in `main.py` deleted; replaced by real Click groups
- All command logic stays in existing `cli/*.py` modules unchanged
- Only `main.py` is substantially rewritten — command registrations and group definitions

## Key Design Decisions

**2-level max depth.** Group + sub-command only. No `load entities download` style 3-part commands. Keeps tab-completion predictable and help pages short.

**`search exclusions` vs `analyze scan-exclusions`.** These are two distinct operations. `search exclusions` is a point lookup — "is UEI ABC123 on the exclusions list?" `analyze scan-exclusions` is a bulk audit — "scan every team member on every active prospect for exclusions." Different inputs, different outputs, different use cases.

**`admin list-org-members` name kept.** Would collide semantically with `prospect list-users` if both were shortened. The admin command lists all members of any org by ID; the prospect command lists users assigned to a specific prospect.

**`load usaspending` replaces `load-transactions`.** The word "transactions" appears in multiple data sources. `usaspending` names the actual source (USASpending.gov), making it unambiguous in `load-history` output.

**`load labor-rates` replaces `load-calc`.** "calc" was an internal abbreviation for the Service Contract Act wage determination tables. `labor-rates` is what the data actually contains.

**No logic moves.** Every command handler function stays in its current `cli/*.py` file. The refactor is entirely in `main.py` registration — `group.add_command(existing_fn, name='new-name')`. This keeps the diff small and the risk low.
